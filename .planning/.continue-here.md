## Critical Anti-Patterns (do NOT repeat these)

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| searchKey collapse on null-flicker | Store sync-wipes subscribed state, then restores the same value; primitive-equal `searchKey` suppresses React dep comparison, so effects that were cancelled during the flicker never re-fire | advisory | Don't wipe-then-restore store state; only wipe when transitioning to a genuinely different context (e.g. different deck) |
| StrictMode double-fire of mount effects | Mount effects in dev run effect → cleanup → effect; requests made inside fire twice even though the first is aborted | advisory | Ref-keyed dedupe (`lastFiredXxxRef`) that no-ops identical re-invocations. Legitimate dep changes produce a new key and fire normally |

<context>
User reported: deck → change commander → back to decks → re-click same deck produced 2 suspicious `/cards/search` requests and an empty card-search section. Surface reading suggested "commander browse" queries; Playwright trace showed they were actually the color-identity auto-search (`id<=wubg+f:commander`) firing twice — the `f:commander` substring made them look browse-ish.

True cause: in `loadForDeck`, `set({ primaryCommander: null, partnerCommander: null, loading: true, ... })` flickered primary to null and back. CardSearchSection's prev-ref effect called `reset()` on the null transition, aborting the in-flight search. When primary restored, `searchKey` (JSON.stringify of filters+identity) was the same string as before — React's Object.is dep comparison saw no change — effect never re-fired. Store stuck at `status:'idle'`, grid empty.

Fix split in two: (1) stop the flicker on same-deck revisit, (2) also dedupe StrictMode's mount double-fire so we don't waste a second request even in the working path. Both regression-tested in isolation — each test verified fails without its respective fix.
</context>
