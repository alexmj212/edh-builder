## Summary

Phase 02.2 stands up Playwright as the project's E2E harness and retroactively covers the six Phase 02.1 UAT flows. CONTEXT.md locks the vast majority of decisions (D-01..D-17): harness choice, folder layout, scripts, route-interception strategy, fixture shape, console policy, and the Phase 3+ standing rule. Research focuses on the narrow set of "Claude's discretion" items (browser coverage, webServer wiring, first-run install, trace/video, tsconfig strategy, ESLint) and on producing concrete code skeletons the planner can slot directly into tasks.

Playwright 1.59.1 is the current stable (verified via `npm view` on 2026-04-12, last modified 2026-04-11). Its APIs for `webServer`, `test.extend`, `page.route`, `page.on('console', ...)`, and `trace: 'retain-on-failure'` are all stable and officially documented. No pre-release surprises.

**Primary recommendation:** Use a **dedicated `tsconfig.e2e.json`** extending `tsconfig.app.json` (keeps `src/` strict rules, adds DOM + Node typings, scopes `@playwright/test` globals to `e2e/`), **chromium-only** project in v1, **auto-applied worker-scoped `stubScryfall` fixture** via `test.extend(..., { auto: true, scope: 'test' })`, **`webServer.command: 'npm run dev'`** with `reuseExistingServer: !process.env.CI`, and **documented one-liner for browser install** (`npx playwright install --with-deps chromium`) — no postinstall hook. Everything else follows CONTEXT.md verbatim.

### Locked Decisions

**Test Harness & Layout**
- **D-01:** Install `@playwright/test` as a devDependency. Use the official test runner — do not wrap it in a custom harness.
- **D-02:** All E2E code lives under `e2e/` at the project root:
  - `e2e/specs/` — `*.spec.ts` files (one file per flow area)
  - `e2e/fixtures/` — Scryfall JSON fixtures (hand-crafted, minimal)

  - `e2e/README.md` — conventions

**Scryfall API Strategy**
- **D-05:** E2E specs **never hit the real Scryfall API**. All `api.scryfall.com/*` traffic intercepted via `page.route()` in a shared helper applied to every spec (`e2e/helpers/stubScryfall.ts`).

  - `/cards/collection` fixtures keyed by oracle_id
- **D-07:** No generic/"browse" search-result fixtures in v1. Each typed search MUST have a corresponding intercepted response.
- **D-08:** Fixtures use real Scryfall JSON shape (per `SCRYFALL_API.md` and `@scryfall/api-types`), not reduced shape.

**Test Isolation & Seeding**
- **D-09:** Each `test()` runs in a fresh browser context (Playwright default) — per-test IndexedDB isolation for free.
- **D-10:** Specs reach starting state **by driving the UI**, not direct Dexie injection. Shared helpers: `createDeck`, `selectCommander`, `selectPartner`, `removePartner`, `changePrimaryCommander`.

- **D-12:** Scryfall interception lives in a **test fixture** (`test.extend`) applied automatically to every spec — no per-spec `beforeEach` boilerplate.

**Console Policy (Cold-Start Smoke)**
- **D-13:** Cold-start smoke attaches `page.on('console', ...)`. **Fails on any `console.error` or `console.warn`** except one allowlisted entry:

| `@playwright/test` | Cypress 13+ | Cypress has a nicer first-time UX but weaker multi-tab/multi-context support, slower CI, and no equivalent of Playwright's per-test `browser.newContext()` isolation. Playwright is the 2024+ consensus choice. |

### Pattern 1: Auto-applied Scryfall stub via `test.extend`

**What:** Wrap every test with automatic `api.scryfall.com/*` interception so no spec ever touches real network.
**When to use:** Always — enforced by D-05/D-12.
**Why auto fixture:** Playwright's auto fixture with `{ auto: true }` runs even when the test doesn't explicitly request it. Pairs perfectly with CONTEXT.md's "no per-spec `beforeEach` boilerplate" requirement.

```typescript
// e2e/helpers/stubScryfall.ts
// Source: https://playwright.dev/docs/test-fixtures (verified 2026-04-12)
import { test as base, type Route } from '@playwright/test';
import thrasios from '../fixtures/cards/thrasios.json' with { type: 'json' };
import tymna from '../fixtures/cards/tymna.json' with { type: 'json' };
import atraxa from '../fixtures/cards/atraxa.json' with { type: 'json' };
import thrasiosSearch from '../fixtures/searches/thrasios-search.json' with { type: 'json' };
import tymnaSearch from '../fixtures/searches/tymna-search.json' with { type: 'json' };
import atraxaSearch from '../fixtures/searches/atraxa-search.json' with { type: 'json' };

const CARDS_BY_ID: Record<string, unknown> = {
  [thrasios.id]: thrasios,
  [tymna.id]: tymna,
  [atraxa.id]: atraxa,
};

// Per-test fixtures (the "TestFixtures" generic)
interface StubFixtures {
  scryfallStub: void; // opaque — auto-applied, no API surface
}

export const test = base.extend<StubFixtures>({
  scryfallStub: [async ({ page }, use) => {
    await page.route(/https:\/\/api\.scryfall\.com\/.*/i, async (route: Route) => {
      const url = new URL(route.request().url());

      // /cards/:id (UUID)
      const idMatch = url.pathname.match(/^\/cards\/([0-9a-f-]{36})$/i);
      if (idMatch && CARDS_BY_ID[idMatch[1]]) {
        return route.fulfill({ json: CARDS_BY_ID[idMatch[1]], status: 200 });
      }

      // /cards/search?q=...
      if (url.pathname === '/cards/search') {
        const q = url.searchParams.get('q') ?? '';
        if (/thrasios/i.test(q)) return route.fulfill({ json: thrasiosSearch, status: 200 });
        if (/tymna/i.test(q))    return route.fulfill({ json: tymnaSearch,    status: 200 });
        if (/atraxa/i.test(q))   return route.fulfill({ json: atraxaSearch,   status: 200 });
        // Unknown query — return empty list to keep tests deterministic (D-07)
        return route.fulfill({
          json: { object: 'list', total_cards: 0, has_more: false, data: [] },
          status: 200,
        });
      }

      // /cards/collection (POST, batch hydration)
      if (url.pathname === '/cards/collection' && route.request().method() === 'POST') {
        const body = route.request().postDataJSON() as { identifiers: Array<{ id?: string; oracle_id?: string }> };
        const data = body.identifiers
          .map(i => (i.id && CARDS_BY_ID[i.id]) || null)
          .filter(Boolean);
        return route.fulfill({
          json: { object: 'list', not_found: [], data },
          status: 200,
        });
      }

      // Unmatched Scryfall path: fail loud so we never silently hit real API
      return route.fulfill({
        status: 599,
        body: `stubScryfall: unhandled route ${route.request().method()} ${url.pathname}`,
      });
    });
    await use();
  }, { auto: true }], // <-- AUTO: applies to every test without import ceremony
});

export { expect } from '@playwright/test';
```

**Usage in spec:**
```typescript
import { test, expect } from '../helpers/stubScryfall'; // re-exported test is auto-stubbed
```

### Pattern 2: Console gate with allowlist (smoke spec only)

**What:** Attach `page.on('console', ...)` before navigation, collect `error`/`warning` messages, assert nothing leaked past the allowlist.
**When to use:** Only in `00-smoke.spec.ts` per D-15.

```typescript
// e2e/helpers/consoleGate.ts
// Source: https://playwright.dev/docs/api/class-consolemessage (verified 2026-04-12)
import type { ConsoleMessage, Page } from '@playwright/test';

interface BenignRule {
  locationIncludes: string;
  textIncludes: string;
  reason: string; // documentation for future maintainers
}

/** Single, named allowlist per D-14. New entries REQUIRE a reason. */
export const KNOWN_BENIGN_CONSOLE: BenignRule[] = [
  {
    locationIncludes: 'storage.ts',        // src/lib/storage.ts:5
    textIncludes:     'Persistent storage not granted',
    reason:           'Expected browser behavior for non-installed/incognito origins (Phase 02.1 UAT test 1)',
  },
];

export interface CapturedMessage { type: string; text: string; location: string }

export function installConsoleGate(page: Page): CapturedMessage[] {
  const unexpected: CapturedMessage[] = [];
  page.on('console', (msg: ConsoleMessage) => {
    const type = msg.type();
    if (type !== 'error' && type !== 'warning') return;
    const location = `${msg.location().url}:${msg.location().lineNumber}`;
    const text = msg.text();
    const isBenign = KNOWN_BENIGN_CONSOLE.some(
      r => location.includes(r.locationIncludes) && text.includes(r.textIncludes),
    );
    if (!isBenign) unexpected.push({ type, text, location });
  });
  return unexpected;
}
```

### Pattern 3: UI-driven flow helpers (role/name locators only)

**What:** Thin wrappers over Playwright locators using accessibility queries per D-11.
**When to use:** Every spec. Specs never call Dexie directly (D-10).

```typescript
// e2e/helpers/commanderFlows.ts
import { type Page, expect } from '@playwright/test';

export async function createDeck(page: Page, name: string): Promise<void> {
  await page.goto('/');
  await page.getByRole('button', { name: 'New Deck' }).first().click();
  await page.getByPlaceholder('Deck name...').fill(name);
  await page.getByRole('button', { name: 'Create' }).click();
  // Click the created deck to navigate into workspace
  await page.getByRole('heading', { name, level: 3 }).click();
  await expect(page).toHaveURL(/\/decks\/\d+$/);
}

export async function selectCommander(page: Page, cardName: string): Promise<void> {
  const primary = page.getByLabel('Search for a commander');
  await primary.fill(cardName);
  await page.getByRole('option', { name: new RegExp(cardName, 'i') }).first().click();
}

export async function selectPartner(page: Page, cardName: string): Promise<void> {
  const partner = page.getByLabel('Search for a partner commander');
  await partner.fill(cardName);
  await page.getByRole('option', { name: new RegExp(cardName, 'i') }).first().click();
}

export async function removePartner(page: Page): Promise<void> {
  await page.getByRole('button', { name: 'Remove partner' }).click();
}

export async function changePrimaryCommander(page: Page, newCardName: string): Promise<void> {
  await page.getByRole('button', { name: 'Change commander' }).click();
  await selectCommander(page, newCardName);
}
```

### Anti-Patterns to Avoid

- **`data-testid` proliferation** — Playwright docs explicitly: *"Prefer user-facing attributes to XPath or CSS selectors."* Every `data-testid` added is a component-side hack the spec now owns. Only introduce one when no accessible name exists AND justified in PR description (per D-11).
- **`page.waitForTimeout(ms)`** — Always flaky. Use web-first assertions (`expect(locator).toBeVisible()`) — they auto-retry until timeout.
- **Shared IndexedDB state between tests** — Playwright's default is fresh `BrowserContext` per test, which gives fresh IDB. Do NOT disable this via `test.use({ storageState })` for these specs.
- **Direct Dexie injection to seed state** — D-10 locks UI-driven setup. Dexie injection masks integration bugs.
- **Generic `*/**` route matchers** — Match Scryfall URLs specifically; a too-broad matcher can intercept Vite's HMR traffic and break the dev server.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dev-server lifecycle for tests | Custom spawn/teardown scripts, `concurrently`, shell-level port checks | Playwright's `webServer` config block | Handles readiness polling, port reuse, stderr forwarding, timeout, SIGTERM on exit. Battle-tested. |
| Per-test browser/IDB isolation | Manual cookie clearing, IDB wipe helpers | Default `BrowserContext` per test | Playwright creates a fresh context automatically — no teardown needed. |
| HTTP mocking in the browser | Fetch monkey-patching, service worker stubs | `page.route()` + `route.fulfill({ json })` | Intercepts at the browser-network boundary; covers `fetch`, `XMLHttpRequest`, and navigation. Typed, stable API. |
| Auto-retry for failing element queries | Custom polling loops, `waitForTimeout` | Web-first assertions (`expect(locator).toHaveText(...)`, `toBeVisible()`) | Built-in retry with actionability checks ("auto-waiting"). |
| Capturing failure artifacts | Custom screenshot/video hooks | `trace: 'retain-on-failure'` + `video: 'retain-on-failure'` | Full DOM/network/console trace per failed test; viewable in `npx playwright show-report`. |
| Test parallelism | Manual worker pools | Playwright's built-in worker model (`workers` config) | Auto-parallelizes across files; each worker gets its own browser context. |
| Console-event plumbing | Wrapping `console.*` in page.addInitScript | `page.on('console', ...)` | Native event hook; captures pre-navigation messages if installed before `page.goto()`. |

**Key insight:** Playwright is a "batteries-included" E2E framework. The instinct to reach for external helpers (msw, jest-extended, custom fixtures) is almost always wrong in this layer — the native API covers the need.

## Common Pitfalls

### Pitfall 1: `reuseExistingServer` in CI

**What goes wrong:** Setting `reuseExistingServer: true` unconditionally lets CI silently run against a stale process from a previous (cached) step, producing ghost failures.
**Why it happens:** Copy-pasted local config.
**How to avoid:** Always `reuseExistingServer: !process.env.CI`. CI gets a fresh server per run; local dev gets instant reuse.
**Warning signs:** Inconsistent CI failures, tests pass locally but fail on PR.

### Pitfall 2: Route fixture registered after first request

**What goes wrong:** If `page.route` is registered inside `beforeEach` but a component fires `fetch` in a top-level module during initial script evaluation, the first request slips through.
**Why it happens:** Fixture setup runs after `page` is created but before `goto`; if `page.route` is awaited in the fixture body, it's live before any navigation.
**How to avoid:** Use the auto-fixture pattern shown above — the `await page.route(...)` in the fixture body runs before the test receives `page`. Never call `page.goto` outside a test body.
**Warning signs:** Flaky tests where the first spec in a file fails but subsequent specs pass.

### Pitfall 3: JSON imports with `assert` vs `with`

**What goes wrong:** TS 5.3+ requires `with { type: 'json' }` (import attributes); older `assert { type: 'json' }` still works in some runtimes but is deprecated and Node 22 LTS may emit warnings.
**Why it happens:** Outdated examples.
**How to avoid:** Use `import foo from '...json' with { type: 'json' };` — TypeScript 6.x supports this natively. Alternative: `import { readFile } from 'node:fs/promises'` + `JSON.parse` inside fixture setup.
**Warning signs:** `ExperimentalWarning: Importing JSON modules is an experimental feature`.

### Pitfall 4: Scryfall search responses not matching actual query

**What goes wrong:** The real `card-search-store` builds a complex query string (`name:thrasios id<=... f:commander`). If `stubScryfall` matches only `q=thrasios`, the real encoded query from the app won't match, and the stub returns an empty list, breaking the spec.
**Why it happens:** Scryfall query builder wraps bare search text in operators.
**How to avoid:** Match on substring patterns (`/thrasios/i.test(q)`) not exact equality. The stub in Pattern 1 above already does this.
**Warning signs:** Spec types "Thrasios" in the partner search box and no result appears in the listbox.

### Pitfall 5: Double-counting Vite HMR traffic

**What goes wrong:** An over-broad `page.route('**/*', ...)` matcher intercepts Vite's `/__vite_ping`, `/@react-refresh`, WebSocket HMR, and the app never loads.
**Why it happens:** Pattern overreach.
**How to avoid:** Match only Scryfall hosts — the regex `/https:\/\/api\.scryfall\.com\/.*/i` in Pattern 1 is correctly scoped. Let everything else fall through to the real dev server.
**Warning signs:** Blank page, console floods with HMR errors.

### Pitfall 6: Cold-start smoke fires before React mounts

**What goes wrong:** `await page.goto('/')` resolves on DOMContentLoaded; React is still hydrating. The console listener misses errors thrown during mount.
**Why it happens:** `page.goto` default `waitUntil` is `'load'`, not `'networkidle'`.
**How to avoid:** Install the console listener BEFORE `page.goto`, then `await expect(page.getByRole('heading', { name: /your decks/i })).toBeVisible()` to wait for the real first paint.

### Pitfall 7: `retries: 2` masking flaky locators

**What goes wrong:** A genuine selector bug gets retried away in CI; passes intermittently.
**Why it happens:** Retries are a safety net, not a correctness fix.
**How to avoid:** Audit `playwright-report/` for any `retry=1` or `retry=2` pass — those are latent flakes to fix, not celebrate. Consider `retries: 1` instead of `2` to keep noise down.

// e2e/specs/02-partner-reload.spec.ts
import { test, expect } from '../helpers/stubScryfall';
import { createDeck, selectCommander, selectPartner } from '../helpers/commanderFlows';

test('partner commander survives hard reload', async ({ page }) => {
  await createDeck(page, 'Reload Test');
  await selectCommander(page, 'Thrasios');
  await selectPartner(page, 'Tymna');

### First-Run Install Strategy

**Recommendation: documented one-liner, NOT a postinstall hook.**

**Why not postinstall:**
- Slows every `npm install` by ~100MB download + system-dep step (sudo on Linux).
- Breaks `npm ci` in contexts that don't need browsers (Docker builds of the app itself, dependency-only CI stages).
- Playwright's own docs treat `install` as a deliberate maintenance task tied to version upgrades, not a package-install side effect: *"every time you update Playwright, you might need to re-run the `install` CLI command."* (Source: https://playwright.dev/docs/browsers)

**Recommended setup (documented in `e2e/README.md`):**

```bash
# First-time setup after clone (one-time per machine):
npm install
npm run e2e:install   # == npx playwright install --with-deps chromium
```

CI runs `npx playwright install --with-deps chromium` as an explicit step (shown in `.github/workflows/e2e.yml` above), so it never relies on postinstall.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Cypress + cy.intercept | `@playwright/test` + `page.route` | 2023–2024 | Playwright now the default E2E pick for new projects; superior multi-context, faster parallelism. |
| Puppeteer + Jest | `@playwright/test` (built-in runner) | 2021+ | Puppeteer is now "Playwright minus the test framework." New projects should not adopt Puppeteer directly. |
| `assert { type: 'json' }` | `with { type: 'json' }` | TS 5.3 (2023) | Import attributes are the standardized form. |
| `actions/setup-node@v3` | `@v5` | 2025 | v5 is current; v3 still works but deprecated. |
| `actions/upload-artifact@v3` | `@v4` | 2024 | v3 sunset 2024; v4 required. |
| Manual webServer spawn | `webServer` config block | Playwright 1.11+ | Native support; no more `start-server-and-test`. |
| `test.describe.serial` for IDB sharing | Fresh context per test (default) | Always | Treat per-test isolation as a feature; don't fight it. |

**Deprecated/outdated:**
- `test.only` and `.skip` committed to main — `forbidOnly: !!process.env.CI` makes CI fail if leftover.
- `page.waitForNavigation()` — replaced by `expect(page).toHaveURL(...)` web-first assertion.
- `page.$()` / `page.$$()` (element handles) — replaced by `page.locator()` / `page.getByRole()`.

## Open Questions

1. **Does Vite's default port (5173) collide with anything else on the dev machine?**
   - What we know: Default is `5173`; Vite auto-increments if taken.
   - What's unclear: `webServer.url: 'http://localhost:5173'` will fail readiness probe if Vite picks `5174`.
   - Recommendation: Pin the port via `vite.config.ts` `server.port: 5173` and `strictPort: true` OR accept the risk and document troubleshooting in `e2e/README.md`. **Planner: prefer pinning — one-line change, eliminates a flake class.**

2. **Does the `CommanderSearch` expose results as `role=option`?**
   - What we know: Grep shows `aria-label` on the input. No `role=option` verification.
   - What's unclear: Whether the result list uses `<li role="option">` or plain `<button>` elements.
   - Recommendation: Planner should have a Wave 0 task that reads `CommanderSearch.tsx` and either (a) confirms role-based locators work, or (b) adjusts helpers to `getByRole('button', { name })` / `getByText(name).click()`. Keep zero component changes in v1.

3. **Does `DeckList` expose the deck row as `role=heading` or `role=button`?**
   - What we know: `<h3>` renders the deck name; the row itself is a `<div onClick>`.
   - What's unclear: Playwright `getByRole('heading')` will find the `<h3>` but clicking the heading may not trigger the row's `onClick` (event doesn't bubble from the heading per DOM).
   - Recommendation: Helper uses `page.locator('div').filter({ has: page.getByRole('heading', { name }) }).click()` OR planner tasks a minimal a11y improvement (wrap row in `<button>` or add `role="button"` + `tabIndex`). **Prefer the locator-first approach** to keep Phase 02.2 zero-source-change.
