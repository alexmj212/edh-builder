# Phase 02.2: Playwright E2E infrastructure and Phase 02.1 backfill — Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers:
1. Playwright installed and configured as the project's E2E harness (headless by default, CI-friendly, `npm run e2e` script).
2. An `e2e/` folder with documented conventions (fixtures, helpers, specs layout).
3. A cold-start smoke spec verifying the app boots with no unexpected console output.
4. Backfill specs covering all six Phase 02.1 UAT flows (partner-slot activation, partner selection, reload-persistence, remove-partner persistence, auto-clear on primary change, primary-clear cascade).
5. A standing rule (documented in the phase summary and referenced in PROJECT/ROADMAP context) that from Phase 3 onward every phase ships E2E coverage before being marked complete.

This phase does **not** add product features, change Dexie schema, or touch the Scryfall client. It installs a test harness and retroactively covers already-shipped functionality.

</domain>

<decisions>
## Implementation Decisions

### Test Harness & Layout

- **D-01:** Install `@playwright/test` as a devDependency. Use the official test runner — do not wrap it in a custom harness.
- **D-02:** All E2E code lives under `e2e/` at the project root:
  - `e2e/specs/` — `*.spec.ts` files (one file per flow area)
  - `e2e/fixtures/` — Scryfall JSON fixtures (hand-crafted, minimal)
  - `e2e/helpers/` — shared helpers (commander selection, deck creation, route-stubbing setup)
  - `e2e/README.md` — conventions (how to add specs, fixture format, when to mock vs hit real)
- **D-03:** Add `npm run e2e` script invoking `playwright test`. Add `npm run e2e:ui` for headed/UI-mode dev runs. Do NOT wire E2E into the default `npm test` (Vitest stays the unit/integration runner).
- **D-04:** Playwright config sets **headless by default**. Reporters: `list` locally, `html` archive for CI. `retries: 2` on CI, `0` locally.

### Scryfall API Strategy

- **D-05:** E2E specs **never hit the real Scryfall API**. All `api.scryfall.com/*` traffic is intercepted via `page.route()` in a shared helper applied to every spec (`e2e/helpers/stubScryfall.ts`).
- **D-06:** Fixtures are **hand-crafted, minimal, and scoped to the exact named cards each spec needs**. At minimum for the 02.1 backfill:
  - Thrasios, Triton Hero (Partner keyword primary)
  - Tymna the Weaver (Partner keyword partner, compatible with Thrasios)
  - Atraxa, Praetors' Voice (non-Partner primary — for auto-clear spec)
  - Search-response fixtures for each of the above (single-card result arrays)
  - `/cards/collection` fixtures keyed by oracle_id for hydration paths
- **D-07:** No generic/"browse" search-result fixtures in v1. Each search typed in a spec MUST have a corresponding intercepted response — if a test needs broader browsing later, add fixtures then.
- **D-08:** Fixtures use the real Scryfall JSON shape (pulled from Phase 2's `SCRYFALL_API.md` notes and the `@scryfall/api-types` package), not a reduced shape. Avoids silent drift from the real API contract.

### Test Isolation & Seeding

- **D-09:** Each `test()` runs in a **fresh browser context** (Playwright default). This gives per-test IndexedDB isolation for free — no explicit teardown of Dexie needed between tests.
- **D-10:** Specs reach their starting state **by driving the UI**, not by direct Dexie injection. Shared helpers encapsulate common flows:
  - `createDeck(page, name)` — create via DeckList UI and land on `/decks/:id`
  - `selectCommander(page, name)` — primary-commander selection via CommanderPanel search
  - `selectPartner(page, name)` — partner-slot selection
  - `removePartner(page)` — click Remove-partner button
  - `changePrimaryCommander(page, name)` — exercise the Change-commander flow
- **D-11:** Helpers are thin wrappers over Playwright locators using **role/name queries** (`getByRole`, `getByLabel`, `getByText`) — no CSS-class or `data-testid` coupling unless a component has no accessible name. If a `data-testid` is introduced for testability, it must be justified in the PR description.
- **D-12:** Scryfall interception is applied in a **test fixture** (Playwright's `test.extend`) so every spec gets it automatically. No per-spec `beforeEach` boilerplate.

### Console Policy (Cold-Start Smoke)

- **D-13:** The cold-start smoke spec attaches a `page.on('console', ...)` listener. It **fails on any `console.error` or `console.warn`** unless the message matches the single allowlisted entry:
  - Source contains `storage.ts:5` AND message contains "Persistent storage not granted"
  - (Phase 02.1 UAT documented this as expected browser behavior for non-installed/incognito origins.)
- **D-14:** The allowlist is a **single, named constant** (`KNOWN_BENIGN_CONSOLE`) in the smoke spec — not spread across specs. Adding future allowlist entries requires an inline comment explaining why the underlying warning is acceptable (or a linked issue to fix it at the source).
- **D-15:** Non-smoke specs do NOT enforce the console policy by default (too noisy during transitional states). Only the dedicated smoke spec is the console gate. If a future spec needs to assert no console output during a specific flow, opt in explicitly.

### Standing Rule for Phase 3+

- **D-16:** Update `.planning/PROJECT.md` (or a new standing-rules section) with: **"From Phase 3 onward, every phase plan must include at least one E2E spec task covering the phase's user-facing acceptance flows before the phase can be marked complete."** The wording should be copy-pasteable into downstream planner prompts.
- **D-17:** The `e2e/README.md` explicitly states this standing rule so contributors (human or agent) see it when opening the folder.

### Claude's Discretion

- **Browser coverage** — not discussed. Pragmatic default: Chromium-only in both `npm run e2e` and CI for v1 speed; multi-browser is a later add when/if cross-browser bugs appear.
- **Dev-server wiring** — use Playwright's `webServer` config block to auto-spawn `npm run dev` (not `vite preview` — dev is what users actually run). `reuseExistingServer: true` locally so an already-running dev server isn't killed.
- **CI workflow file** — add a GitHub Actions `.github/workflows/e2e.yml` as part of this phase (uses `actions/setup-node`, installs Playwright browsers, runs `npm run e2e`). Keeps "CI-ready" from being theoretical.
- **Fixture file naming / oracle_id constants** — Claude decides naming and helper shapes during planning.
- **How the `e2e` script handles first-run browser download** — use `npx playwright install --with-deps chromium` either as a postinstall hook or documented one-liner in `e2e/README.md`.
- **Trace/video capture policy** — enable `trace: 'retain-on-failure'` and `video: 'retain-on-failure'` as reasonable defaults.

## Existing Code Insights

### Reusable Assets
- **`@scryfall/api-types`** already a dep — fixtures should `satisfies` the exported `ScryfallCard` / `List<ScryfallCard>` types for compile-time shape checking.
- **Vite dev server** is the canonical run target (`npm run dev`). Playwright's `webServer` block reuses it rather than running `vite preview` on a built bundle (closer to what users see).
- **Existing accessible labels/roles** from Phase 02 UI (CommanderPanel, DeckWorkspace) — helpers can query by role/name with no test-id retrofits expected.

### Established Patterns
- **Strict TypeScript** — `e2e/` uses the same strictness. `tsconfig.json` may need an include addition (or a dedicated `tsconfig.e2e.json`).
- **Co-located tests** in `src/**` — E2E deviates by living under `e2e/` at project root. Document the distinction in `e2e/README.md`.
- **Vitest is the unit/integration runner** — Playwright is additive; neither replaces the other.
- **No ESLint rules currently exclude `e2e/`** — may need an ESLint override to allow Playwright's `test.describe`/`test` globals or rely on imports.

### Integration Points
- `package.json` — new scripts (`e2e`, `e2e:ui`), new devDep (`@playwright/test`).
- `vite.config.ts` — no changes expected; Playwright runs against the normal dev server.
- `.gitignore` — add `playwright-report/`, `test-results/`, `e2e/.auth/` (if any) so CI artifacts don't pollute the tree.
- `.github/workflows/e2e.yml` — new workflow; project has no CI yet, this establishes it.
- `.planning/PROJECT.md` or standing-rules doc — add the Phase 3+ E2E-coverage rule.

<specifics>
## Specific Ideas

- **Realism over speed for 02.1 backfill** — the whole point is to prove the UAT flows work in a real browser. UI-driven setup (D-10) is non-negotiable even though Dexie-injection would be faster.
- **Fixture contract matches Scryfall's real shape** (D-08) — use `@scryfall/api-types` so fixtures fail at compile time if the real shape drifts.
- **Single allowlisted console line, named constant** (D-13/14) — prevents allowlist creep.
- **Helpers query by role/name, not test-ids** (D-11) — keeps specs aligned with accessibility and avoids test-only component changes.
- **Standing rule is documented, not just mentally agreed to** (D-16/17) — Phase 3 planner will pick it up from PROJECT.md / `e2e/README.md`.

</specifics>
