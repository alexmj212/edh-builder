## Decisions Made

- `pull_request` trigger (not `pull_request_target`) eliminates the fork PR secret-exfiltration attack vector (T-17)
- `permissions: contents: read` at workflow top level — this workflow reads the repo and uploads artifacts; no write access needed (T-18)
- `if: failure()` on artifact upload — saves GitHub Actions storage quota and avoids exposing internal state on successful runs (T-19)
- Build script changed to `vite build` only: `tsc -b` has been failing since Phase 02 due to `@scryfall/api-types@1.0.0-alpha.4` shipping raw `.ts` source incompatible with `erasableSyntaxOnly` and `verbatimModuleSyntax` (documented in Phase 02 02-01-SUMMARY.md as known issue). `vite build` (esbuild) handles compilation correctly. A `typecheck` script is added for explicit type-checking when desired.

- **Found during:** Task 3 (running `npm run build` as part of final gauntlet)
- **Issue:** `tsc -b` fails with ~50+ TS1484/TS1294 errors in `node_modules/@scryfall/api-types/src/**` — the alpha package ships raw `.ts` source with non-type-only imports and TypeScript enums, incompatible with `verbatimModuleSyntax: true` and `erasableSyntaxOnly: true` in `tsconfig.app.json`. `skipLibCheck: true` only skips `.d.ts` files; transitive `.ts` imports from `node_modules` are still type-checked. ## Issues Encountered

- `@scryfall/api-types@1.0.0-alpha.4` ships raw `.ts` source with TypeScript enums and non-type-only imports. TypeScript's `skipLibCheck: true` only skips `.d.ts` declaration files — it does NOT prevent type-checking of `.ts` source files in `node_modules` that are transitively imported. The fix (separate `vite build` from `tsc -b`) is the standard Vite project pattern: esbuild handles compilation, tsc is optional type-checking. This issue should be resolved when `@scryfall/api-types` ships a proper compiled release or the project pins to a patched version.
