# Phase 02.2 Code Review

## Summary

The Phase 02.2 Playwright harness is well-structured and demonstrates careful
thinking about test isolation, stub realism, and accessibility-first locators.
The architecture — auto-applied stub fixture, default-deny unmatched routes, typed
fixtures, role/name locators, and per-spec IDB validation — is sound and follows
Playwright best practices closely. The console gate is correctly scoped to the
cold-start smoke spec, its allowlist is minimal and justified, and the security
posture of the CI workflow is tight.

Three warnings are worth resolving before the harness grows. The most impactful is
that `npm run build` no longer invokes `tsc -b` (changed to silence
`@scryfall/api-types` alpha source errors in `node_modules`), and the CI workflow
has no `typecheck` step, so future type regressions in `src/` will not block
builds or green CI. The second is that the `build` script change was made as an
auto-fix without pinning the underlying root cause (the alpha package) or
configuring `skipLibCheck` correctly. The third is that the raw IDB introspection
block is copy-pasted identically across four spec files, creating a maintenance
hazard as the schema evolves. The four informational notes cover a `console.warn`
escape path in non-smoke specs, a subtle stub-search false-positive risk,
unverified GitHub Actions version tags, and the `/cards/collection` stub branch
that handles `oracle_id`-based lookups silently as not-found.

## Findings

### WR-01 Warning: `npm run build` silently drops TypeScript type-checking; no `typecheck` step in CI

**File:** `package.json` (build script); `.github/workflows/e2e.yml`
**Severity:** warning

**Issue:** The `build` script was changed from `tsc -b && vite build` to
`vite build` to work around `@scryfall/api-types@1.0.0-alpha.4` shipping raw
TypeScript source (not pre-compiled declarations), which causes `tsc -b` to fail
on that package's own TS2484 errors even with `skipLibCheck: true` (because
`skipLibCheck` skips `.d.ts` files, not `.ts` source files in `node_modules`).
A separate `typecheck` script (`tsc -b`) was added, but it is not referenced in
the only CI workflow (`e2e.yml`). The result: CI only runs the Playwright suite
and never type-checks `src/`. A future contributor introducing a type error in
application source will see a green CI run.

**Recommendation:** Add a dedicated typecheck CI job or step. The correct long-term
fix is to add `"skipLibCheck": true` to `tsconfig.app.json` in a way that also
covers imported `.ts` source — or, preferably, replace the alpha package with a
stable release or a local type-only declaration shim. A tactical workaround is to
add `paths` aliasing in tsconfig to redirect the import to a hand-written `.d.ts`.
In the short term, add a `typecheck:app` step to `e2e.yml` (or a separate
`typecheck.yml` workflow) so type regressions in `src/` are caught:

```yaml
- name: Type-check application source
  run: npm run typecheck:e2e
  # Also add: npm run typecheck once the @scryfall/api-types issue is resolved
```

At minimum, the CI workflow should run `npm run typecheck:e2e` so e2e helper
type errors are caught in the pipeline.

### WR-02 Warning: IDB introspection block duplicated verbatim across four specs

**File:** `e2e/specs/03-partner-reload.spec.ts:22-38`, `04-partner-remove.spec.ts:26-40`,
`05-partner-autoclear.spec.ts:24-37`, `06-primary-clear-cascade.spec.ts:25-37`
**Severity:** warning

**Issue:** All four specs that validate IndexedDB state contain a near-identical
`page.evaluate()` block that opens the `'EDHBuilder'` IndexedDB, reads the
`decks` object store, and filters by deck name. If the database name, store name,
or access pattern changes, four files need updating. The deck name used as the
lookup key (`'Reload Test'`, `'Remove Test'`, etc.) is also the only thing
differentiating the four copies, making it easy to miss a future edit.

**Recommendation:** Extract a helper to `e2e/helpers/commanderFlows.ts` (or a
new `e2e/helpers/idbHelpers.ts`):

```typescript
export async function getDeckRowByName(
  page: Page,
  name: string,
): Promise<Record<string, unknown> | null> {
  return page.evaluate(async (deckName: string) => {
    const req = indexedDB.open('EDHBuilder');
    const db = await new Promise<IDBDatabase>((resolve, reject) => {
      req.onsuccess = () => resolve(req.result);
      req.onerror  = () => reject(req.error);
    });
    const allReq = db.transaction('decks', 'readonly').objectStore('decks').getAll();
    const rows = await new Promise<Array<Record<string, unknown>>>((resolve, reject) => {
      allReq.onsuccess = () => resolve(allReq.result as Array<Record<string, unknown>>);
      allReq.onerror   = () => reject(allReq.error);
    });
    db.close();
    return rows.find(r => r.name === deckName) ?? null;
  }, name);
}
```

Each spec then becomes a single `await getDeckRowByName(page, 'My Test Deck')` call.

### WR-03 Warning: `console.warn` from partner hydration failure is invisible to non-smoke specs

**File:** `src/store/commander-store.ts:57`; `e2e/helpers/consoleGate.ts`
**Severity:** warning

**Issue:** `commander-store.ts` emits a `console.warn` when partner hydration fails
during `loadForDeck`. The console gate (`installConsoleGate`) is only installed in
`00-smoke.spec.ts`. Specs 01-06 do not install it, so a broken partner hydration
path that emits this warning would produce a passing green test suite. The README
documents this as intentional ("Other specs do not enforce the console policy
unless they opt in explicitly"), but the warning is specifically about an
application data-loss scenario (partner silently dropped) that the partner-reload
spec (03) is designed to catch.

**Recommendation:** Either install `consoleGate` in the reload and remove specs
(03, 04) where hydration failure is a realistic failure mode, or promote
`installConsoleGate` to an auto-fixture in `stubScryfall.ts` so it applies to
every spec automatically. The auto-fixture approach is lower overhead and
consistent with the existing `scryfallStub` pattern:

```typescript
// In stubScryfall.ts, add consoleGateFixture: [async ({ page }, use) => {
//   const unexpected = installConsoleGate(page);
//   await use();
//   expect(unexpected, ...).toHaveLength(0);
// }, { auto: true }]
```

If the gate is kept opt-in, at minimum add it to specs 03 and 04 where the
hydration warning would indicate a real regression.

### IN-01 Info: Stub search matching uses substring regex — may return Thrasios for a query that mentions "thrasios" incidentally

**File:** `e2e/helpers/stubScryfall.ts:54-56`
**Severity:** info

**Issue:** The `/cards/search` stub matches the query parameter `q` against
`/thrasios/i`, `/tymna/i`, and `/atraxa/i` as substring patterns. The real app
builds queries like `(t:legendary t:creature or o:"can be your commander") f:commander name:thrasios`,
so the match works correctly today. However, if the search query ever contains
a card name as oracle text (e.g., `o:"thrasios"`) or a filter argument that
happens to include one of the three fixture names, the stub would return the
wrong fixture. This is a low-risk issue given the narrow fixture set, but the
matching is not anchored to the `name:` field.

**Recommendation:** No immediate action required. If the fixture set grows,
consider matching on `name:thrasios` substrings specifically:
`/\bname:thrasios\b/i.test(q)` or using a URL parameter parse approach.

### IN-02 Info: `/cards/collection` stub does not handle `oracle_id`-based identifiers — would silently return empty `data`

**File:** `e2e/helpers/stubScryfall.ts:63-74`
**Severity:** info

**Issue:** The collection endpoint stub resolves identifiers only by `i.id`. The
Scryfall API also supports `oracle_id` in the identifiers array. If the application
ever calls `/cards/collection` with `{ oracle_id: "..." }` identifiers (not `id`),
the stub would silently return an empty `data` array (passing the null filter on
line 69) rather than the HTTP 599 deny response. The app currently has no
`/cards/collection` call in `scryfall-client.ts`, so this is preemptive
infrastructure, but the silent mismatch would be confusing when the endpoint is
eventually used.

**Recommendation:** Add a 599 response for the case where an identifier is provided
but cannot be resolved, or add a comment flagging that `oracle_id` lookups are
not currently supported by the stub:

```typescript
// NOTE: oracle_id-based identifiers are not resolved; add oracle_id lookup
// here when the app starts using collection with oracle_id identifiers.
```

### IN-03 Info: GitHub Actions version tags are not commit-SHA pinned

**File:** `.github/workflows/e2e.yml:26-27`
**Severity:** info

**Issue:** The workflow uses `actions/checkout@v5`, `actions/setup-node@v5`, and
`actions/upload-artifact@v4`. Tag-based action references are mutable — a
compromised action maintainer could push a new commit to the `v5` tag and
execute arbitrary code in CI. For public repositories, the recommended security
posture is to pin to a specific commit SHA (e.g., `actions/checkout@a5ac7e51b41094c92402da3b24376905380afc29`).
Additionally, `actions/checkout@v5` and `actions/setup-node@v5` may be
hypothetical future versions — as of early 2026 the latest stable tags for those
actions were `v4`. If `v5` does not exist on GitHub, this workflow would fail on
the first run.

**Recommendation:** Verify the `v5` tags exist for `actions/checkout` and
`actions/setup-node`. If they do not, downgrade to `v4`. For security hardening,
pin all three actions to their full commit SHAs and add a comment with the
resolved tag for readability:

```yaml
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
```

### IN-04 Info: `tsconfig.e2e.json` disables `erasableSyntaxOnly` and `verbatimModuleSyntax` without documentation

**File:** `tsconfig.e2e.json:8-10`
**Severity:** info

**Issue:** `tsconfig.e2e.json` overrides two strict compiler flags inherited from
`tsconfig.app.json`: `verbatimModuleSyntax: false` and `erasableSyntaxOnly: false`.
These are intentional overrides — Playwright's test runner compiles specs
differently from Vite's bundler mode — but the file has no comment explaining why
the overrides are necessary. A future maintainer who tightens `tsconfig.app.json`
may inadvertently re-enable them and introduce friction.

**Recommendation:** Add a brief comment explaining the rationale:

```jsonc
// verbatimModuleSyntax and erasableSyntaxOnly are disabled here because
// Playwright compiles specs through its own bundler (not Vite), which does not
// require these constraints. The JSON import assertions (with { type: 'json' })
// in stubScryfall.ts also require this relaxation.
```

## Positive notes

**Default-deny stub design.** The HTTP 599 catch-all in `stubScryfall.ts` is the
right decision. It means an unhandled Scryfall route (e.g., a new API call added
to the app without a corresponding stub entry) causes a loud, obvious test failure
rather than a real network hit or a silent empty response. This is the correct
behavior for a hermetic test harness.

**Compile-time fixture shape checking.** Using `satisfies FixtureCardShape` on
every imported JSON fixture is an unusually careful approach. It catches fixture
drift (e.g., a field renamed in `@scryfall/api-types`) at `npm run typecheck:e2e`
time rather than at test runtime. The workaround for the discriminated-union
limitation (using a minimal structural interface instead of `ScryfallCard.Any`
directly) is well-commented and correct.

**Fixture IDs are deterministic synthetic UUIDs.** Using `00000000-0000-4000-8000-00000000000N`
instead of real Scryfall IDs prevents any risk of accidentally matching production
data and makes the stub lookup table trivially auditable.

**IDB assertions validate persistence, not just UI state.** Specs 03-06 do not
stop at asserting the visible card image — they open IndexedDB directly and assert
on the persisted field values. This is the correct level of coverage for a
persistence phase; it catches bugs where the UI updates optimistically but the
Dexie write was dropped.

**Role/name locators throughout.** All helpers in `commanderFlows.ts` use
`getByRole`, `getByLabel`, and `getByText`. There is not a single `data-testid`
or CSS selector in the helper layer. This makes the locators resilient to
styling changes and doubles as an accessibility audit: if the role or label is
wrong in the component, the test will fail.

**Auto-fixture pattern eliminates per-spec boilerplate.** The `{ auto: true }`
option on `scryfallStub` means every spec that imports `test` from
`'../helpers/stubScryfall'` gets network interception applied without a
`beforeEach` call. This is exactly the pattern described in Playwright's
documentation and prevents the common mistake of forgetting to install the stub
in a new spec.

**Concurrency + `forbidOnly` in CI.** Setting `workers: 1` in CI avoids IndexedDB
isolation surprises on shared runners, and `forbidOnly: !!process.env.CI`
prevents `test.only` leakage. Both are the recommended production CI settings.
