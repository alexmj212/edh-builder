## Scryfall API Strategy

### How should E2E specs handle Scryfall API traffic?

| Option | Description | Selected |
|--------|-------------|----------|
| page.route() with hand-crafted fixtures | Intercept api.scryfall.com/*, return minimal JSON fixtures. Fast, deterministic, CI-safe. | ✓ |
| Real Scryfall API with rate-limit respect | Hit real API — realistic but flaky, slow, blocked in offline CI | |
| Record-and-replay (HAR or custom) | Record once, replay — larger payloads, more machinery up front | |

**User's choice:** page.route() with hand-crafted fixtures
**Notes:** Rationale: E2E reliability + CI safety dominate. Minimal-maintenance fixture set matches the small number of named cards specs reference.

### Are partner backfill fixtures scoped to specific named cards, or also cover search UI (typeahead results list)?

| Option | Description | Selected |
|--------|-------------|----------|

| Broader: include representative search-result pages | Realistic browsing + heavier fixture upkeep | |

**User's choice:** Minimal: targeted named-card fixtures only
**Notes:** Defer broader fixtures until a spec actually needs generic browsing.
