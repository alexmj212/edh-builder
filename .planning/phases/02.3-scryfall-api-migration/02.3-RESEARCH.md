### `MagicPageResult<T>` (pagination)

```ts
class MagicPageResult<T> {
  get count(): number;        // total_cards
  get hasMore(): boolean;     // whether another page exists
  all(): Promise<T[]>;        // fetches every page until !hasMore ‚Äî DANGEROUS for `f:commander`, tens of thousands of rows
  get(limit: number): Promise<T[]>;  // fetches pages until `limit` is reached
  next(): Promise<T[]>;       // fetches next page, bumps internal page counter
  page(n: number): Promise<T[]>;     // jump to specific page
}
```

**Key behavior:** Each `MagicPageResult` is a new instance with its own mutable
page counter. After the first `.next()`, `hasMore` / `count` are populated.
The library internally increments `query.page += 1` before each fetch.

**Gotcha ‚Äî no `next_page` URL:** Our current `ScryfallList.next_page` URL is
not exposed. We must carry the `MagicPageResult` reference itself across
`search` ‚Üí `loadMore`, or re-call `.page(n+1)` with the same query.

### `Card` type

```ts
type Card = CardCore & CardPrint & GameplayCard;
```

**Crucially: NOT a discriminated union on `layout`** ‚Äî it's a flat intersection
of three interfaces. The `layout` field is on `CardCore` as `CardLayout` (a
string literal union of 24 values), but there's no narrowing needed. This
**eliminates the `satisfies ScryfallCard.Any` ‚Üí JSON literal-widening problem**
that drove `tsconfig.e2e.json` to disable strict flags.

## AbortSignal Integration Path

### Finding: NO native AbortSignal support

Verified by inspecting `dist/scryfall-api.js`:
- `CardApi.search` returns `MagicPageResult` synchronously ‚Äî no options object
  with `signal`.
- `fetcher()` (lines 95-134) builds an `init` with `method` and optional
  `body`/`headers` ‚Äî no `signal` threaded in.
- `simpleFetcher = createDebounceFetcher(createRetryFetcher())` ‚Äî the retry
  wrapper re-calls on 5xx, so even if we injected a signal, the library would
  not honor an abort between retries.
- No `fetch` override API is exposed to the consumer. `createDebounceFetcher`
  accepts `fetcher = fetch` as a default parameter, but that's internal.

### Recommendation: D-04 option 3 (promise-race wrapper)

Option 1 (native) and option 2 (fetch injection) are both off the table. Option
3 is the only viable path. This was explicitly anticipated by CONTEXT.md D-04
and labeled acceptable for a browser SPA.

**Single-point wrapper** (pseudocode, lives in `src/lib/scryfall.ts`):

```ts
import { Cards, type Card, type CardSearch, MagicPageResult } from 'scryfall-api';

class AbortError extends DOMException {
  constructor() { super('Aborted', 'AbortError'); }
}

function abortable<T>(promise: Promise<T>, signal?: AbortSignal): Promise<T> {
  if (!signal) return promise;
  if (signal.aborted) return Promise.reject(new AbortError());
  return new Promise<T>((resolve, reject) => {
    const onAbort = () => reject(new AbortError());
    signal.addEventListener('abort', onAbort, { once: true });
    promise.then(
      v => { signal.removeEventListener('abort', onAbort); resolve(v); },
      e => { signal.removeEventListener('abort', onAbort); reject(e); },
    );
  });
}

export async function fetchCardById(id: string, signal?: AbortSignal): Promise<Card> {
  const card = await abortable(Cards.byId(id), signal);
  if (!card) throw new Error(`Card not found: ${id}`);
  return card;
}

// Pagination wrapper ‚Äî keeps MagicPageResult alive between search/loadMore.
export interface SearchResult {
  data: Card[];
  hasMore: boolean;
  totalCards: number;
  _page: MagicPageResult<Card>;  // opaque handle for next-page fetch
}

export async function searchCards(
  query: string,
  options?: CardSearch,
  signal?: AbortSignal,
): Promise<SearchResult> {
  const page = Cards.search(query, options);
  const data = await abortable(page.next(), signal);
  return { data, hasMore: page.hasMore, totalCards: page.count, _page: page };
}

export async function fetchNextPage(
  handle: SearchResult,
  signal?: AbortSignal,
): Promise<SearchResult> {
  const data = await abortable(handle._page.next(), signal);
  return { data, hasMore: handle._page.hasMore, totalCards: handle._page.count, _page: handle._page };
}
```

**Semantics trade-off (already accepted by CONTEXT.md D-04):**
The underlying `fetch` completes in the background. The caller sees the
rejection immediately on abort; the network round-trip continues, consumes the
next rate-limit slot (100ms), and the result is discarded. Acceptable for a
browser SPA where the observable behavior (stale results discarded) is what
SRCH-05 actually requires.

**Mitigation for background-request noise:**
- Our current `card-search-store.ts` already uses a module-level
  `AbortController` that replaces itself on each search ‚Äî the race is at the
  store layer, not the fetch layer. That contract is preserved verbatim.
- No additional logging or warning required ‚Äî matches CONTEXT.md spec.

**D-08 conclusion:** PASS via option 3.

## Rate-Limit Throttle Semantics

### Finding: module-global, hard-coded 100ms, NOT configurable

From `dist/scryfall-api.js` lines 20-44:

```js
// src/fetcher/debounce.fetcher.ts
var lastQuery = 100;         // module-scoped
var rateLimit = 100;          // module-scoped, NOT exported
async function sleep(ms = 0) { /* setTimeout */ }
async function debounce() {
  const now = Date.now();
  const timeSinceLastQuery = now - lastQuery;
  if (timeSinceLastQuery >= rateLimit) {
    lastQuery = now;
  } else {
    const timeUntilNextQuery = rateLimit - timeSinceLastQuery;
    lastQuery += timeUntilNextQuery;  // reserve the slot BEFORE sleeping
    await sleep(timeUntilNextQuery);
  }
}
function createDebounceFetcher(fetcher = fetch) { /* ... */ }

// src/fetcher/fetcher.ts
var simpleFetcher = createDebounceFetcher(createRetryFetcher());  // module singleton
```

### Properties

| Property | Value | D-09 implication |
|---|---|---|
| Scope | **Module-global** ‚Äî `lastQuery` and `rateLimit` are module-level `var`s. Every `Cards.search()` / `Cards.byId()` / `Sets.all()` / etc. funnels through the single `simpleFetcher` instance. | ‚úÖ SHARED ‚Äî meets D-01/D-09 requirement. Two concurrent stores (commander-store + card-search-store) share one throttle. |
| Minimum gap | **Hard-coded 100ms.** Not exported, not configurable via constructor, not mutable via any public API. | ‚úÖ Exactly matches SRCH-06 and our current client's `MIN_GAP_MS = 100`. No behavior change. |
| Configurability | None. The `rateLimit` variable is closed over ‚Äî we cannot adjust it without patching the library or monkey-patching at module load (not recommended). | ‚öÝ Neutral ‚Äî we don't want to change it anyway. |
| Reservation semantics | The throttle reserves the next slot **before** sleeping (`lastQuery += timeUntilNextQuery`). This means N queued calls are spaced 100ms apart deterministically, even if they all fire at `t=0`. Our current client has the same behavior. | ‚úÖ Equivalent to ours. |
| Abort-cancellation | The `sleep(ms)` in `debounce()` is NOT abort-aware. A signal aborting while waiting for the next slot does NOT cancel the sleep. Our current `waitForRateLimit` IS abort-aware. | Minor regression ‚Äî a user who types 10 chars fast will consume 10 rate-limit slots sequentially in the background, even after the `AbortController` fires. Acceptable per D-04's "underlying fetch completes in background" stance. |

### Singleton import pattern (what the planner must enforce)

Because throttling is module-scoped, the wrapper module (`src/lib/scryfall.ts`)
MUST be the **only** code that imports from `scryfall-api`. If two modules
independently `import { Cards } from 'scryfall-api'`, they still share the
singleton (ES modules are cached by URL), so correctness is preserved ‚Äî but the
architectural rule "one entry point" still applies for testability.

**D-09 conclusion:** PASS. Drop our `waitForRateLimit` / `lastRequestTime` /
`__resetRateLimit`. The library handles it identically, module-globally,
sharedly.

**Test-cleanup note:** Without an exposed `__resetRateLimit`, Vitest tests
cannot force the throttle back to zero between runs. In practice this only
matters if tests assert on wall-clock timing. Recommendation: use
`vi.useFakeTimers()` and `vi.advanceTimersByTime(100)` in the 2-3 tests that
care; do NOT attempt to reach into the library's private state.

## Fixture & Type-Cast Implications

### Why casts exist today (per CONTEXT.md and 02.2-CONTEXT D-05‚Ä¶D-08)

`ScryfallCard.Any` from `@scryfall/api-types` is a discriminated union over
`layout: 'normal' | 'split' | 'flip' | ...`. JSON imports widen string literals
to `string`, so `satisfies ScryfallCard.Any` fails. The workaround: an
intermediate `FixtureCardShape` structural type + `as unknown as ScryfallCard.Any`
at the lookup table, plus disabling `erasableSyntaxOnly` / `verbatimModuleSyntax`
in `tsconfig.e2e.json`.

### `scryfall-api`'s `Card` is NOT a discriminated union

`type Card = CardCore & CardPrint & GameplayCard` ‚Äî a flat intersection.
`CardCore.layout: CardLayout` is a literal union, but it is not a narrowing
discriminator (no `Card<'normal'>` subtype is ever used ‚Äî every member of the
intersection contributes fields unconditionally).

### Implication for `satisfies Card`

Two cases:

**Case A ‚Äî `satisfies Card` works directly (optimistic):**
If TypeScript is happy to widen `"normal"` ‚Üí `CardLayout` during the `satisfies`
check (since `CardLayout` is a union of literals that includes `"normal"`), the
casts disappear entirely:

```ts
// After migration ‚Äî no intermediate shape, no `as unknown as`
import thrasios from '../fixtures/cards/thrasios.json' with { type: 'json' };
const _thrasios = thrasios satisfies Card;  // works if TS widens correctly
```

Some fixture fields are likely missing vs. the full `Card` (our fixtures only
contain fields the app reads ‚Äî not `prices`, `purchase_uris`, `related_uris`,
`finishes`, `border_color`, etc.). `Card` has many required fields from
`CardPrint`, so **`satisfies Card` will likely fail on completeness**, not on
`layout`.

**Case B ‚Äî `satisfies` still fails on missing required fields (realistic):**
Since `CardPrint` requires ~30 fields our fixtures don't have (`border_color`,
`finishes`, `booster`, `frame`, `games`, etc.), the fixtures need either:

1. **Expansion** ‚Äî add the missing required fields (tedious but accurate).
2. **Structural subtype** ‚Äî define a `FixtureCard = Pick<Card, 'id' | 'oracle_id' | ...>` and `satisfies FixtureCard`. Same pattern as today but over the new type.
3. **Relaxed cast** ‚Äî `fixture as Card` (loses compile-time safety ‚Äî not recommended).

**Recommendation:** Case B option 2 ‚Äî define a project-local
`type FixtureCard = Pick<Card, ...fields the app reads>` in `stubScryfall.ts`,
`satisfies FixtureCard` for each fixture, and cast the lookup map entries
`as Card` (single, localized cast at the boundary, justified by a one-line
comment).

This eliminates:
- `as unknown as ScryfallCard.Any` (becomes `as Card` ‚Äî single level, not double-unknown).
- `FixtureCardShape` placeholder ‚Üí replaced by `Pick<Card, ...>` which stays in sync with the library type.
- `verbatimModuleSyntax: false` / `erasableSyntaxOnly: false` overrides in `tsconfig.e2e.json` ‚Äî those existed because `@scryfall/api-types` ships raw `.ts` source that violates `erasableSyntaxOnly`; `scryfall-api` ships pre-compiled `.d.ts`, so the overrides can be deleted.

### Playwright fixture JSON is unchanged

The hand-crafted JSON bodies in `e2e/fixtures/cards/*.json` and
`e2e/fixtures/searches/*.json` represent real Scryfall responses. Their shape is
dictated by the API, not by any TS library. **No content changes needed.** Only
the TS type alias that validates them changes.

### Edge case ‚Äî library auto-parses dates

The `fetcher()` at `dist/scryfall-api.js:89-94` walks JSON with
`JSON.parse(content, dateParser)`. The `dateParser` regex matches
`^\d{4}-\d{2}-\d{2}?$` and converts matches to `Date`. Affected fields:
`released_at`, `Set.released_at`, ruling `published_at`, preview `previewed_at`.
**Fixtures returned via `page.route().fulfill({ json: ... })` don't go through
this parser** ‚Äî Playwright's stub returns the Response, but the library still
runs `response.text()` ‚Üí `JSON.parse(..., dateParser)`. So when the wrapper
returns a card from a stubbed search, any `released_at` string **will** be
parsed to `Date`. If downstream code reads `card.released_at` as a string, it
breaks. **None of our current code does.** Phase 3 BUILD-08 will need to handle
this. Flag for the Phase 3 planner.

## Open Questions for the Planner

1. **Wrapper module name** ‚Äî `src/lib/scryfall.ts` (fresh) vs. `src/lib/scryfall-client.ts` (repurposed). Research has no opinion; ROADMAP allows both. **Recommendation: fresh `scryfall.ts`** ‚Äî shorter, no legacy baggage, matches how we'd name a new file today. Delete `scryfall-client.ts` entirely.

2. **`SearchResult._page` handle shape** ‚Äî The opaque `MagicPageResult` reference inside `SearchResult` needs a TS access modifier decision. Options:
   - Public field named `_page` with leading underscore (convention: "internal")
   - Symbol-keyed field (true private)
   - A discriminated shape returned only from `searchCards` and accepted by `fetchNextPage`
   **Recommendation: public `_page: MagicPageResult<Card>` with a JSDoc `@internal`** ‚Äî simplest, matches our existing module style.

3. **Should `getImageUri` move into the wrapper module or its own module?** ‚Äî Currently a helper in `scryfall-client.ts`. It doesn't depend on `scryfall-api`; it's a pure function over the `Card` shape. **Recommendation: stays in the wrapper module** (`src/lib/scryfall.ts`) because it's semantically "Scryfall card image extraction" and all its consumers (CommanderSearch, CommanderPanel, future CardResultCell) already import from the wrapper.

4. **Should we export `Card` from the wrapper module as a re-export, or have consumers import from `scryfall-api` directly?** ‚Äî **Recommendation: re-export.** `import type { Card } from '../lib/scryfall'` is the single import point; swapping the underlying library in the future (unlikely but possible) touches only the wrapper. Also makes the testing strategy (`vi.mock('../lib/scryfall')`) cleaner.

5. **Spike exit artifact format** ‚Äî CONTEXT.md says `02.3-01-SUMMARY.md` on BLOCK; should we also write the summary on PASS? **Recommendation: yes**, write the summary unconditionally. On PASS it documents verified decisions (throttle is shared, no abort support, dates auto-parsed, etc.) and live-probe output. Informs 02.3-02‚Ä¶05.

6. **CI typecheck step placement** ‚Äî New job in `e2e.yml` before the Playwright job, vs. separate `typecheck.yml` workflow. **Planner-discretion.** Single-workflow with typecheck as an early step is slightly faster (shared setup) and gives a clear ordered failure signal.
