# Code Review — Phase 02.3: scryfall-api Migration

Reviewed all 31 listed files as a unit. No critical bugs found. The migration
is functionally sound. Four warnings require attention before or shortly after
this phase closes; six info items are improvements or clarifications with no
blocking impact.

---

## Overall Assessment

The migration from the hand-rolled `scryfall-client.ts` to `scryfall-api@^4.0.5`
is well-executed. The wrapper in `src/lib/scryfall.ts` correctly preserves all
four D-series design decisions (AbortSignal via promise-race, shared throttle
via library internals, opaque pagination handle, Zod validation boundary). The
query builders in `src/lib/scryfall-queries.ts` are pure, testable, and handle
user input correctly. The type sweep is clean: zero `@scryfall/api-types` or
`ScryfallCard` references remain anywhere in `src/` or `e2e/`. Production code
has zero `as any` / `as never` casts; test-only casts are acceptable and
localized. The CI pipeline ordering is correct (typecheck before Playwright),
and `tsconfig.app.json` has no residual `exclude` entries.

---

## Strengths

**Wrapper correctness.** `toSearchResult` reads `page.hasMore` and `page.count`
_after_ `await page.next()` returns. The library updates those fields inside
`next()`, so callers always see post-fetch values. The ordering is correct and
confirmed against the library source (`scryfall-api/dist/scryfall-api.js`
lines 172–183).

**AbortSignal semantics preserved.** `abortable()` correctly handles three
cases: pre-aborted signal (immediate rejection), mid-flight abort (event-driven
rejection), and no signal (pass-through). The listener is cleaned up in both
settlement paths, preventing memory leaks.

**Injection safety.** `encodeFragment` strips all `"` characters before
wrapping multi-word fragments in quotes, making the query builders safe against
user input that contains double-quote characters. All user-controlled inputs
(name, type, oracleText, nameFragment) pass through this function before
reaching `Cards.search`.

**Dead code fully removed.** `scryfall-client.ts` and its test file are deleted
and all import sites updated. No lingering `ScryfallCard` or
`@scryfall/api-types` references remain.

**CI ordering.** `Typecheck (src + e2e)` runs before `Install Playwright
browsers` and `Run Playwright suite` in `.github/workflows/e2e.yml`. Type errors
fail fast without burning Playwright minutes.

**tsconfig.app.json clean.** No `exclude` array is present; the temporary
exclusion added in plan 02.3-04 was correctly removed in plan 02.3-05.

**E2E fixture shape.** The `card-search-sample.json` and
`card-search-sample-page2.json` fixtures include all fields the Zod boundary
validates (`id`, `name`) and all fields the UI reads (`type_line`,
`oracle_text`, `mana_cost`, `image_uris`, `color_identity`). The `has_more`
flag round-trips correctly: page 1 has `true`, page 2 has `false`, matching
the Load More visibility assertions.

## Issues Found

### Warnings

**WR-1 — `validatedData` discards the Zod-parsed output; the cast is redundant if schema strengthens**

File: `src/lib/scryfall.ts`, lines 46–49

```ts
function validatedData(raw: unknown[]): Card[] {
  CardArraySchema.parse(raw);   // return value discarded
  return raw as Card[];         // cast the original, not the parsed result
}
```

`CardArraySchema.parse(raw)` is called purely for its side-effect (throwing
`ZodError` on invalid data). If the schema ever uses `.transform()` or `.strip()`
instead of `.passthrough()`, the validated output would differ from `raw` and
the cast would silently return un-transformed data. As written with `.passthrough()`
this is not a bug, but it is a latent trap for the next schema evolution. The
pattern should use the Zod output:

```ts
function validatedData(raw: unknown[]): Card[] {
  const parsed = CardArraySchema.parse(raw);
  return parsed as Card[];
}
```

This makes the code's dependency on `passthrough()` explicit and safe to
tighten later.

**WR-2 — `searchPartnersFor` 'none' case allocates a real `MagicPageResult` with empty query; `Cards.search('')` would HTTP-error if `next()` were ever called**

File: `src/lib/scryfall.ts`, lines 107–113

```ts
const emptyPage = Cards.search('', { order: 'edhrec' });
return { data: [], hasMore: false, totalCards: 0, _page: emptyPage };
```

The `_page` slot must hold a `MagicPageResult<Card>` for the interface to
typecheck, so some handle is required. However, calling `Cards.search('')`
constructs a handle that would send a bad-request query to Scryfall if
`next()` were ever invoked on it. The returned `SearchResult` has
`hasMore: false`, so the `loadMore` guard in `card-search-store.ts` will never
reach `fetchNextPage` with this handle in the normal execution path — the risk
is currently contained. But if `CommanderSearch` or any future caller stores
this result and accidentally calls `fetchNextPage`, the library would fire a
live HTTP request to `api.scryfall.com/cards/search?q=` and silently return
empty data (Scryfall returns 400 → library returns `undefined` → `next()`
returns `[]`).

No test covers the "accidentally call fetchNextPage on a 'none' handle" path.
Consider replacing the empty-string query with a sentinel that can never
accidentally succeed, or document the invariant with an assertion:

```ts
// Option A: use an object cast that satisfies the type but marks it as inert
const emptyPage = { next: () => Promise.resolve([]), hasMore: false, count: 0 } as unknown as MagicPageResult<Card>;

// Option B: guard at fetchNextPage (already implicit via hasMore:false in the store guard)
// -- add an explicit assert here if preferred
```

**WR-3 — No test for the Zod validation boundary rejecting malformed API responses**

Files: `src/lib/scryfall.ts`, `src/lib/scryfall.test.ts`

`validatedData` is the D-05 Zod validation boundary and is called on every
successful `next()` response. There is no test that verifies it throws when
data is malformed (e.g., a card object missing the `id` field, or a non-array
response). The existing tests supply well-formed data and only verify the
happy path.

A minimal gap-fill:

```ts
it('searchCards rejects when API returns cards missing required id field', async () => {
  const page = makePageStub([{ name: 'No ID' }] as never, false, 1);
  searchSpyRef.current.mockReturnValue(page);
  await expect(searchCards('test')).rejects.toThrow(); // ZodError
});
```

Without this test, a malformed API response would propagate as a generic
`ZodError` to callers, which the store catches as `status: 'error'`. While
that is acceptable runtime behavior, the absence of a test means a future
schema change (e.g., removing `.passthrough()`) could silently discard valid
cards instead of throwing.

**WR-4 — `encodeFragment` regex includes `"` in the wrapping-decision test after already stripping all quotes**

File: `src/lib/scryfall-queries.ts`, lines 7–11

```ts
export function encodeFragment(v: string): string {
  const cleaned = v.replace(/"/g, '').trim();
  if (!cleaned) return '';
  return /[\s"()]/.test(cleaned) ? `"${cleaned}"` : cleaned;
  //             ^ this `"` can never match — quotes were stripped above
}
```

After `replace(/"/g, '')`, `cleaned` cannot contain `"`. The `"` in the
test regex is dead code. It creates a false impression that quotes in the
cleaned string would trigger wrapping. The correct regex is `/[\s()]/`.

This is a readability defect, not a runtime bug. The fix is one character:

```ts
return /[\s()]/.test(cleaned) ? `"${cleaned}"` : cleaned;
```

The accompanying tests do not cover this case (quoting a string that contains
only `"` characters) and would continue to pass after the fix.

**IR-2 — `hasOracleId` guard in `card-cache.ts` is overly defensive given `Card.oracle_id` is non-optional in the resolved type**

File: `src/lib/card-cache.ts`, lines 8–13

```ts
function hasOracleId(c: Card): c is Card & { oracle_id: string } {
  return 'oracle_id' in c && typeof (c as ...).oracle_id === 'string' && ...
}
```

`Card = CardCore & CardPrint & GameplayCard`. `CardPrint.oracle_id` is
`string` (non-optional, non-nullable), making `Card.oracle_id` required in
the intersection. The narrowing predicate is therefore always `true` for a
valid `Card` value. The guard remains harmless as defensive code, but the
`Card & { oracle_id: string }` return type annotation is redundant — the
compiler already knows `Card` has `oracle_id: string`. A comment explaining
why the guard exists (e.g., "runtime safety for potentially partial objects
arriving from untrusted cache reads") would clarify the intent.

**IR-4 — `stubScryfall.ts` uses a single `asCard` escape hatch but the `pickKeys` compile-time drift check does not verify field _values_, only field _presence_**

File: `e2e/helpers/stubScryfall.ts`, lines 32–42

The `pickKeys` array ensures `keyof FixtureCard` stays in sync at compile
time. But `FixtureCard` uses `Pick<Card, ...>` which inherits the library's
type widths. For example, `color_identity: Color[]` in `Card` is a strict
`Color[]` (where `Color = 'W' | 'U' | 'B' | 'R' | 'G'`). The JSON import
widens the literal strings to `string[]`. The `asCard` cast absorbs this
widening silently.

This is acceptable because the Zod boundary in `scryfall.ts` validates real
API responses at runtime, and the fixtures are only used in E2E stubs (never
processed through Zod since they bypass the network). The risk is that a
fixture with an invalid `color_identity` value (e.g., `"Z"`) would not be
caught. The comment in the file already documents this trade-off accurately.
No action required unless fixtures are reused in unit tests.

**IR-5 — `partner-detection.ts` `detectPartnerType` regex can capture trailing whitespace in the named partner name**

File: `src/lib/partner-detection.ts`, line 33

```ts
const namedMatch = oracle.match(/Partner with ([^\n(]+)/i);
if (namedMatch) {
  return { kind: 'named', partnerName: namedMatch[1].trim() };
}
```

The capture group `[^\n(]+` is greedy and stops at newline or `(`. For
oracle text like `"Partner with Virtus the Veiled\n(Reminder text)"`, the
capture is `"Virtus the Veiled"` (clean). For `"Partner with Sidar Kondo of
Jamuraa (When this creature enters)"`, the capture stops at `(` and is
`"Sidar Kondo of Jamuraa "` — trailing space removed by `.trim()`. Correct
in all known cases. The `buildPartnerQuery` then calls `.replace(/"/g, '')`
on the trimmed name before embedding it in `!"${name}"`. This chain is safe
and all test cases pass.

Noted because the regex `[^\n(]+` does allow other trailing punctuation (e.g.,
a period from a rule line ending). Real Scryfall card data does not have this,
but no assertion guards against it in tests.

**IR-6 — `CardSearchSection` first `useEffect` runs on every render (no dependency array) to detect commander change**

File: `src/components/CardSearchSection.tsx`, lines 61–66

```ts
useEffect(() => {
  if (prevPrimaryOracleId.current !== primaryOracleId) {
    prevPrimaryOracleId.current = primaryOracleId;
    reset();
  }
});  // no dependency array
```

This is a deliberate choice: a `[primaryOracleId]` deps array would call
`reset()` on the initial mount when `primaryOracleId` is `null`, which is
unwanted. The no-deps pattern correctly skips the reset on mount by comparing
to the ref initialized with the same value. The logic is correct.

The concern is that this effect runs after every render of `CardSearchSection`,
including unrelated state changes (e.g., a filter keystroke). The guard
`prevPrimaryOracleId.current !== primaryOracleId` makes it a no-op in those
cases, but it adds a comparison on every render. A more idiomatic alternative
is to initialize the ref with a sentinel (e.g., `useRef<string | null | undefined>(undefined)`) to distinguish "first render" from "null commander":

```ts
const prevPrimaryOracleId = useRef<string | null | undefined>(undefined);
useEffect(() => {
  if (prevPrimaryOracleId.current === undefined) {
    prevPrimaryOracleId.current = primaryOracleId;
    return; // skip reset on mount
  }
  if (prevPrimaryOracleId.current !== primaryOracleId) {
    prevPrimaryOracleId.current = primaryOracleId;
    reset();
  }
}, [primaryOracleId, reset]);
```

This would allow adding a proper dependency array and satisfy the exhaustive-deps
lint rule without calling `reset()` on mount. Not a regression risk at current
scale but worth addressing before additional renders are added to this component.

## Recommendations

1. **WR-1 (low-effort):** Replace `CardArraySchema.parse(raw); return raw as Card[]` with `return CardArraySchema.parse(raw) as Card[]` so the Zod output is actually used. One-line fix.

2. **WR-2 (low-effort):** Replace `Cards.search('', ...)` with an inert object cast in the `searchPartnersFor` 'none' branch, or add a code comment explicitly stating that `fetchNextPage` must never be called on this handle (the `hasMore: false` guard is the enforcer).

3. **WR-3 (medium-effort):** Add one negative test to `scryfall.test.ts` that feeds a card without an `id` field through `searchCards` and asserts that the returned promise rejects with a `ZodError`.

4. **WR-4 (trivial):** Remove the `"` from the `encodeFragment` wrapping test regex: `return /[\s()]/.test(cleaned) ? ...`.

5. **IR-6 (future cleanup):** Refactor the first `useEffect` in `CardSearchSection` to use an explicit deps array with a sentinel-initialized ref. Prioritize before adding more state to this component.

## Performance Notes

- The `useMemo` for `identity` in `CardSearchSection` is appropriate — it only
  recomputes when `primary` or `partner` changes.
- The `filterKey = JSON.stringify(...)` memoization is correct but `JSON.stringify`
  on every render (before memoization) adds minor overhead. Acceptable at this
  object size.
- The library's 100ms module-level throttle (`var lastQuery = 100` initialized
  to 100ms from epoch, meaning the very first request fires without waiting)
  is shared across all `Cards.*` calls in the process. This is correct D-09
  behavior. No wrapper interference.
- `cacheCards` is called with `void` in both `search` and `loadMore` — fire and
  forget. If the Dexie write fails, the error is silently swallowed. This is
  intentional (cache is best-effort) but a `console.warn` on failure would aid
  debugging in production.
