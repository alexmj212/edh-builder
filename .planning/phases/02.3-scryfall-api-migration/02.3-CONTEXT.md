# Phase 02.3: Scryfall client migration to `scryfall-api` — Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the hand-rolled `src/lib/scryfall-client.ts` and the stale
`@scryfall/api-types@1.0.0-alpha.4` types with the actively-maintained
`scryfall-api` (MarioMH8) package. Restore `tsc -b && vite build` in the build
pipeline, drop the `verbatimModuleSyntax: false` / `erasableSyntaxOnly: false`
overrides in `tsconfig.e2e.json`, and eliminate the `as unknown as
ScryfallCard.Any` casts scattered across the codebase. Closes REVIEW.md WR-01
from Phase 02.2.

The phase has **BLOCK authority** in 02.3-01 (spike & go/no-go gate). If
browser/Vite compat, AbortSignal support, or `Card` field parity cannot be
resolved, the spike aborts the migration and a fallback path is taken (see
Claude's Discretion — fallback-if-BLOCK deferred to spike-summary).

This phase does **not** add product features, change Dexie schema, change
Zustand store semantics, or modify user-facing UI. It is a typed-client
replacement with build-pipeline cleanup.

</domain>

<decisions>
## Implementation Decisions

### Wrapper Boundary & Scope

- **D-01 (Rate limit):** Adopt `scryfall-api`'s throttle. Delete the hand-rolled
  `waitForRateLimit` / `lastRequestTime` module state. Configure the library's
  throttle to match the current 100ms minimum-gap behavior that satisfies
  SRCH-06. Spike (02.3-01) must confirm the throttle is **shared across all
  callers**, not per-instance — if it's per-instance, this decision reverts to
  "keep ours" and the spike-summary documents why.

- **D-02 (Query builders — primary rule):** Where `scryfall-api` exposes a
  query-builder helper that cleanly covers one of our builders, **use theirs**.
  Candidates include a general search-query builder (today: `buildSearchQuery`
  with `name`, `type`, `oracleText`, `colorIdentity`, `f:commander`) and a
  commander-name fragment builder (today: `buildCommanderSearchQuery`).

- **D-03 (Query builders — fallback):** For queries the library does not cover,
  keep a small local module `src/lib/scryfall-queries.ts` containing only the
  uncovered builders. The 4-variant partner search (`generic Partner`,
  `friendsForever`, `chooseBackground`, `named partner-with`) is the most
  likely candidate to live here — it is unusually project-specific and
  unlikely to be a generic library concern. Do **not** inline partner-query
  logic at the call site (CommanderSearch.tsx) — keep it in a module to
  preserve testability and to match the existing module-per-concern pattern.

- **D-04 (AbortSignal):** If `scryfall-api` does not accept `AbortSignal`
  natively, wrap each call in a **thin promise-race wrapper** that listens on
  the signal and throws `AbortError` when aborted. The underlying fetch may
  still complete in the background — acceptable trade-off for a browser SPA
  where the user-observable behavior (stale results discarded) matches
  SRCH-05. Preferred ordering during planning:
    1. Check if `scryfall-api` accepts `AbortSignal` directly → use it.
    2. Check if it allows a custom `fetch` injection → inject one that
       forwards our signal.
    3. Fall back to the promise-race wrapper.
  Whichever path lands, it must be a **single wrapper** that all call sites
  go through — no ad-hoc per-call abort plumbing.

- **D-05 (Zod validation):** Keep Zod response-envelope validation at the
  project's boundary (our new client wrapper). TypeScript types are
  compile-time; Zod is the runtime safety net. Established memory:
  stubbed-response tests can pass while the real API drops fields —
  compile-time typing alone doesn't catch that. `ScryfallListSchema` (or its
  equivalent) stays. If `scryfall-api` ships matching Zod schemas, prefer
  theirs; otherwise keep ours.

### Spike Gate (02.3-01) — Go/No-Go Acceptance

The spike task has BLOCK authority. It passes if ALL of the following hold,
and documents any gaps inline in `02.3-01-SUMMARY.md`:

- **D-06:** `scryfall-api` installs and imports cleanly in Vite dev + build
  (no Node-only deps, no CommonJS-only shape that breaks ESM interop).
- **D-07:** `Card` type parity — every field our 11 usage sites read
  (`oracle_id`, `id`, `name`, `color_identity`, `keywords`, `oracle_text`,
  `type_line`, `card_faces`, `image_uris`, `legalities`, plus partner-related
  fields) is present on the new `Card` type or a documented subtype.
- **D-08:** `AbortSignal` path is resolved — either native support, custom
  fetch injection, or the wrapper design from D-04 is demonstrably viable.
- **D-09:** Rate-limiter behavior satisfies D-01 (shared, configurable to
  100ms) OR we consciously revert D-01.
- **D-10 (gap tolerance):** If the library covers MOST but not ALL of the
  above, proceed and document gaps in `02.3-01-SUMMARY.md`. Query-builder
  coverage gaps are explicitly ALLOWED — D-03 covers that case. Do NOT raise
  the bar to "spike BLOCKs on any gap."

### Claude's Discretion

- **Wrapper module naming and location** — Whether the new code lives in
  `src/lib/scryfall-client.ts` (repurposed) or a fresh `src/lib/scryfall.ts`
  (deleting the old file entirely) is planner-discretion. The ROADMAP
  deliverable says "deleted (or converted to a thin wrapper)" — both are OK.
- **Exact signature of the thin wrapper** — shape of `searchCards`,
  `fetchCardById`, etc. can match the current exports or mirror
  `scryfall-api`'s surface; planner picks based on blast radius across the 11
  usage sites.
- **Fallback-if-BLOCK plan** — If 02.3-01 BLOCKs, the recovery plan (local
  `.d.ts` shim vs. Zod-derived types vs. abandon) is decided at spike-exit,
  not here. The spike-summary proposes, the user ratifies, then a new phase
  or amended plan executes it.
- **Test + CI closeout details** — Fixture evolution approach in
  `e2e/helpers/stubScryfall.ts` (regenerate vs. adapt casts), specific
  Playwright regression spec shape per 02.2 standing rule, and whether 02.3
  also closes WR-01's CI-side recommendation (adding a typecheck step to
  `.github/workflows/e2e.yml`) are planner-discretion. Default: 02.3 SHOULD
  close the full WR-01 — both the build script restoration AND the CI
  typecheck step — since the root cause (`@scryfall/api-types` alpha) is
  being removed in this same phase.

## Existing Code Insights

### Reusable Assets
- **`ScryfallListSchema` (Zod)** — the current Zod envelope in
  `scryfall-client.ts`. Keep and reuse per D-05 at the new wrapper boundary.
- **`getImageUri(card, size)`** — pure helper over the `Card` shape
  (`image_uris` / `card_faces[0].image_uris` fallback). Survives the
  migration regardless of library choice.
- **`encodeFragment(v)`** — local string-quoting helper used by query
  builders. Moves to `scryfall-queries.ts` if that module is created
  (per D-03).
- **`detectPartnerType(card)`** (partner-detection.ts) — consumed by
  `searchPartnersFor`. Already pure; only its type parameter changes.
- **E2E fixtures in `e2e/fixtures/*.json`** — hand-crafted Scryfall JSON
  bodies. Content-stable across the library swap; only the TS type that
  validates them changes.

### Established Patterns
- **Zustand stores are the consumer boundary** — `commander-store` and
  `card-search-store` are where `scryfall-client` is imported. Keeping the
  import surface stable minimizes blast radius across the stores.
- **AbortController created module-level in `card-search-store`** (Phase 2
  decision — not in Zustand state, to avoid re-render on abort). The D-04
  abort wrapper must interoperate with this pattern, not require re-plumbing.
- **Zod schemas validate at IO boundaries** (DB reads, API reads).
  Matches D-05.
- **Pure-function library modules** (partner-detection, card-cache) —
  accept `ScryfallCard.Any` as input. Migration changes only the imported
  type alias.
- **Tests co-located with source** (`*.test.ts` next to each module). New
  wrapper tests follow the pattern.
- **E2E fixtures type-checked via `satisfies FixtureCardShape`** — casts
  exist because `ScryfallCard.Any` is a discriminated union on `layout`.
  Post-migration, the `Card` type from `scryfall-api` may or may not be a
  similar union — fixtures' type-assertion strategy may simplify.

### Integration Points
- `package.json` dependencies — `scryfall-api` added, `@scryfall/api-types`
  removed.
- `package.json` scripts — `build` restored to `tsc -b && vite build`.
- `tsconfig.e2e.json` — `verbatimModuleSyntax` / `erasableSyntaxOnly`
  overrides removed (they exist today solely to work around the alpha
  package).
- `.github/workflows/e2e.yml` — add `npm run typecheck` step (closes WR-01's
  CI half per Claude-discretion default above).
- 11+ import sites updated from `import type { ScryfallCard } from
  '@scryfall/api-types'` to the `scryfall-api` equivalent (likely
  `import type { Card } from 'scryfall-api'` — confirm in spike).

<specifics>
## Specific Ideas

- **Rate-limit throttle must be shared, not per-instance** (D-01) — if
  `scryfall-api` creates a new throttle per client instance, two concurrent
  search flows (commander search + card search) could each burst their own
  allocation. Spike verifies this or D-01 reverts.
- **Partner search logic stays project-local** (D-03) — the 4-variant branch
  (`generic` / `friendsForever` / `chooseBackground` / `named`) is too
  specific to belong in a generic library.
- **Zod is kept even if library ships types** (D-05) — runtime validation is
  the point; TS types already exist today via the alpha package and still
  haven't prevented shape drift. Recent feedback memory
  (`feedback_verify_external_apis`) is exactly this concern.
- **Abort wrapper is single-point, not scattered** (D-04) — whichever of the
  three abort paths wins, all call sites go through one wrapper. No per-call
  ad-hoc `Promise.race`.
- **WR-01 closes fully** (Claude's Discretion default) — root cause (alpha
  package) is removed in 02.3; closing only the build-script half would
  leave the CI typecheck gap open for no reason.

</specifics>
