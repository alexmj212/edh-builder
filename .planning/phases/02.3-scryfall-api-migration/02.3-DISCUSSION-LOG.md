## Wrapper Boundary & Scope — Question Batch 1

### Q1: If scryfall-api provides its own rate-limiter, how do we handle our 100ms shared throttle (SRCH-06)?

Options:
- **Adopt theirs (Recommended)** — Delete waitForRateLimit; configure theirs
  to match 100ms. Requires spike confirmation that throttle is shared.
- **Keep ours, layer on top** — Safer if theirs is per-instance, but risks
  double-delay.
- **Depends on spike** — Defer to 02.3-01 findings.

**User selected:** Adopt theirs (Recommended).
**Captured as:** D-01.

### Q2: Query builders — where do they live?

Options:
- **Keep local module (Recommended)** — Extract to scryfall-queries.ts.
- **Use scryfall-api's query helpers if available** — Delete ours if their
  coverage is sufficient.
- **Inline per call site** — No builder abstraction.

**User selected:** Use scryfall-api's query helpers if available.
**Captured as:** D-02 (primary rule — prefer theirs); drove follow-up Q5 for
fallback behavior.

### Q3: AbortSignal propagation — what if scryfall-api doesn't forward signals?

Options:
- **Thin wrapper that races signal vs. promise (Recommended)** — Promise.race
  against AbortSignal. Library completes in background; callers see
  cancellation.
- **Require first-class support or BLOCK** — Strictest.
- **Pass fetch via library's hook if available** — Cleaner; depends on
  extensibility.

**User selected:** Thin wrapper that races signal vs. promise (Recommended).
**Captured as:** D-04 (with a preferred ordering: native → custom fetch →
wrapper — preserving the spirit of the other two options as earlier
preferences).

### Q4: Zod response-envelope validation — keep or trust library types?

Options:
- **Keep it at our boundary (Recommended)** — Runtime safety net; stubbed
  tests pass while real API drops fields.
- **Drop Zod, trust types** — Simpler.
- **Zod only on cache hydration** — Middle ground.

**User selected:** Keep it at our boundary (Recommended).
**Captured as:** D-05.
