## Plan Interpretation Notes (value-level `satisfies` omitted)

Plan Task 2 §interfaces spec shows:

```ts
const _thrasios = thrasios satisfies FixtureCard;
```

This form does NOT compile because JSON module imports widen string literals (`"normal"` → `string`, `["U","G"]` → `string[]`), so they cannot satisfy Card's literal unions (`CardLayout`, `Color[]`, `Legality`). Attempted intermediate forms (`Omit<FixtureCard, 'layout'>`, `Partial<FixtureCard>`) were all blocked by the same widening on `color_identity`.

**Resolution:** The `Pick<Card, ...>` FixtureCard alias is KEPT as documentation-of-intent and referenced through `pickKeys: readonly (keyof FixtureCard)[]` as a compile-time key-drift guard (if Card drops any enumerated field, the array type narrows). `asCard` accepts `object` (which every JSON import trivially satisfies) and performs the single `as unknown as Card` cast. Runtime safety is provided by the Zod schema at `src/lib/scryfall.ts` (validates real API responses) plus the default-deny 599 route handler in the stub (unstubbed endpoints fail the E2E run fast).

This preserves all the acceptance grep constraints:
- `Pick<Card,` present (in the FixtureCard alias)
- `const asCard` present
- Exactly 1 `as unknown as` (in asCard's return expression)
- 3 `asCard(` call sites (the CARDS_BY_ID entries)
- 0 references to `@scryfall/api-types`, `ScryfallCard`, or `FixtureCardShape`

## Process Gap: vitest runner vs tsc -b

The 12 latent test-file errors (Deviation 1) are a process finding, not just a fix: **vitest's default runner uses esbuild's transpile-only mode and never performs TS type-checking**. Type errors in test files land silently — tests pass, CI appears green, the bug stays latent until a full `tsc -b` sweep. The CI typecheck step added in Task 3 closes this loop going forward: every PR now runs `npm run typecheck` before any test or build step, so future drift between vitest's runtime behavior and TypeScript's type view is caught at PR time.

Memory cross-reference: [feedback_test_first_workflow](feedback_test_first_workflow.md) — "review code AND tests together". This plan's SUMMARY is the first evidence of tests-and-code reviewed as one unit surfacing type bugs the test runner alone can't catch.
