## Warnings

### WR-01: CardResultCell container is focusable with `cursor-pointer` but has no activation handler

**File:** `src/components/CardResultCell.tsx:58-62`
**Issue:** The outer `<div>` has `tabIndex={0}`, a visible focus ring, and `cursor-pointer` styling — all signals to users (keyboard and mouse) that the element is interactive. However, only the inner "+" `<button>` handles activation. A keyboard user who tabs onto the card and presses Enter/Space gets no feedback; a mouse user who clicks the card art (not the + button) also gets nothing. The `focus-within` overlay reveal works, so the tab stop is justified for disclosure — but the affordances overstate what activation exists.

This is a Phase 03.1 change (per the e2e spec comment "preserved tab stop from Plan 03"), so the pattern is intentional, but the `cursor-pointer` on line 60 is misleading given there's no click handler on the container.

**Fix:** Either (a) remove `cursor-pointer` from the outer container (keep it on the inner `+` button only) so cursor affordance matches reality, or (b) add an activation handler that forwards Enter/Space/click on the container to `onAdd` when `!disabled`:
```tsx
// Option A — recommended, lowest-risk:
<div
  data-testid="card-result-cell"
  className="relative rounded-lg overflow-hidden bg-surface border border-border group focus:outline-none focus:ring-2 focus:ring-accent"
  tabIndex={0}
>
// (remove `cursor-pointer` from outer; inner button keeps its own cursor classes)
```

### WR-02: `DeckWorkspace` can briefly render "Deck not found" before `loadDecks` resolves

**File:** `src/components/DeckWorkspace.tsx:15-46`
**Issue:** On first visit to `/decks/:id`, the effect on lines 15-19 triggers `loadDecks()` only if `decks.length === 0`. Between mount and the async resolution, if `useDeckStore().loading` is not yet `true` (e.g., the store defaults `loading: false` until the thunk flips it on the next tick), `decks.find(...)` returns `undefined` and the "Deck not found" branch renders — then flickers to the real workspace once decks populate. The deck-store's thunk timing governs whether this actually happens in practice.

**Fix:** Gate "not found" on a positive "we've loaded at least once" signal rather than "the array is empty right now":
```tsx
const hasLoaded = useDeckStore(s => s.hasLoaded); // add this flag to deck-store
// …
if (loading || !hasLoaded) {
  return <div className="text-text-secondary text-center py-16">Loading deck...</div>
}
const deck = Number.isNaN(numericId) ? undefined : decks.find(d => d.id === numericId)
if (!deck) { /* show not-found */ }
```
Or, cheaper: keep `loading` gating but ensure `loadDecks()` sets `loading: true` synchronously before its first `await`. Verify against deck-store to confirm whether this flicker is actually reachable — if not, downgrade to Info.

## Info

### IN-01: Redundant `!disabled` guard in CardResultCell click handler

**File:** `src/components/CardResultCell.tsx:81`
**Issue:** `onClick={() => { if (!disabled) onAdd(); }}` — the `disabled` HTML attribute on line 78 already prevents the browser from dispatching click events on the button. The inline guard is dead defensive code.
**Fix:** Simplify to `onClick={onAdd}`. (The CardResultCell.test.tsx test "does not call onAdd when disabled" passes `pointerEventsCheck: 0` to bypass userEvent's own disabled check — it verifies the browser semantics, not the JS guard.)

### IN-02: Non-null assertion on `primaryCommander!` in CommanderPanel

**File:** `src/components/CommanderPanel.tsx:153`
**Issue:** `primaryForPartner={primaryCommander!}` uses a non-null assertion. The surrounding `partnerSlotActive` branch (line 103) guarantees `!!primaryCommander`, so the assertion is sound, but it's fragile to future refactors of the gating condition.
**Fix:** Hoist the narrowed reference into a local:
```tsx
) : !partnerCommander ? (
  (() => {
    // primaryCommander is guaranteed non-null here via partnerSlotActive
    const primary = primaryCommander;
    if (!primary) return null;
    return (
      <>
        <EmptyArt label="Partner (optional)" />
        <div className="mt-2">
          <CommanderSearch mode="partner" primaryForPartner={primary} onSelect={…} />
        </div>
      </>
    );
  })()
)
```
Or keep the assertion and add an `// eslint-disable-next-line @typescript-eslint/no-non-null-assertion` with a short rationale comment — the status quo is acceptable.

### IN-03: `DeckColumn` fetch effect depends on `lookupMap` and recreates AbortController on each resolution

**File:** `src/components/DeckColumn.tsx:54-73`
**Issue:** The effect re-runs whenever `lookupMap` changes (i.e., every successful fetch batch). On the re-run, `missing` is empty so the async IIFE returns early — but the `AbortController` is still created and then `.abort()`'d by the cleanup on the next run. This is harmless (no leak, no error) but wasteful and mildly obscures intent.
**Fix:** Split the dependency — read `lookupMap` via a ref for the `missing` computation, or early-return before creating the controller:
```tsx
useEffect(() => {
  const ids = Array.from(new Set(cards.map(c => c.scryfallId)));
  const missing = ids.filter(id => !lookupMap.has(id));
  if (missing.length === 0) return;
  const controller = new AbortController();
  // …
  return () => controller.abort();
}, [cards, lookupMap]);
```
Order-of-operations change only — correctness unchanged.

### IN-04: Test helper type drift — `fakeCard` returns `Record<string, unknown>` or `any` inconsistently

**File:** `src/components/CardResultCell.test.tsx:6-23`, `src/components/CommanderPanel.test.tsx:9-11`, `src/components/DeckColumn.test.tsx:12-24`
**Issue:** Each test file redefines its own `fakeCard`/`makeCard` helper with a different signature — `Record<string, unknown>`, `any`, or a narrowly-typed overrides param. Callers then `as any`-cast at every render site. This duplication is easy to drift and undercuts the type safety the `Card` type is supposed to provide.
**Fix:** Extract a shared typed factory in `src/test-utils/fakeCard.ts` that returns `Card`:
```ts
import type { Card } from '../lib/scryfall';
export function fakeCard(overrides: Partial<Card> = {}): Card {
  return { id: 'c-1', name: 'Fake', type_line: 'Instant', keywords: [], color_identity: [], image_uris: { normal: 'x', art_crop: 'y' }, …overrides } as Card;
}
```
Callers drop the `as any` noise entirely. Purely a test-quality improvement; out-of-phase-scope but worth tracking.
