<objective>
Fix UI-REVIEW finding #2: DeckColumn's view-switch scroll reset targets the wrong DOM node. The inner `DeckColumn` root has `overflow-y-auto` declared but the true scroll container is the outer `flex-[2]` wrapper in `DeckWorkspace.tsx:55` (which has `max-h-[calc(100vh-6rem)] overflow-y-auto`). At deck sizes that overflow the outer container, `scrollRef.current.scrollTop = 0` silently no-ops.

Resolution strategy (locked — see Approach below): lift the scroll-reset target to the outer wrapper. DeckWorkspace owns a `useRef<HTMLDivElement>` pointed at the outer `flex-[2]` div; DeckColumn accepts a new optional `onViewToggle?: () => void` prop that DeckWorkspace wires to reset `outerRef.current.scrollTop = 0`. Additionally, remove the redundant `overflow-y-auto` from DeckColumn's own root div so there's a single scroll context (resolves the "double scroll wrapper" advisory in UI-REVIEW Pillar 5).

Purpose: Close the highest-impact functional gap in UI-REVIEW (Pillar 6) and the double-scroll spacing concern (Pillar 5) in one coordinated change.

Output:
- `src/components/DeckWorkspace.tsx` owns an outer-wrapper ref and passes `onViewToggle` callback
- `src/components/DeckColumn.tsx` drops `overflow-y-auto` from root, drops internal `scrollRef`, invokes `props.onViewToggle?.()` inside `handleToggle`
- `src/components/DeckColumn.test.tsx` gains a regression test asserting `onViewToggle` is called when the view toggle is clicked
</objective>
