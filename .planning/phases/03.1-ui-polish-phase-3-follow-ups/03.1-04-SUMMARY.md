1. **`CommanderPanel "Change commander" button shows focus ring on keyboard focus`**

   - Regression guard: `toHaveAttribute('tabindex', '0')` — pins Plan 03's preserved tab stop (pairs with focus-within overlay reveal).
   - Focuses, asserts `toBeFocused()`, same class-surface + computed-box-shadow assertions as Test 1.
   - Console-gate.

3. **`view toggle resets outer scroll container scrollTop when deck overflows`**
   - Locates the outer wrapper via `.locator('[data-testid="deck-column"]').locator('..')` — the `flex-[2]` div in DeckWorkspace that owns `outerScrollRef`.
   - Injects a 2000px spacer as the wrapper's first child and sets `scrollTop = 500` to synthesise overflow.
   - Clicks `Grid` in the Deck-view ViewToggle group.
   - `expect.poll(...).toBe(0)` on the wrapper's scrollTop — proves `handleViewToggle → outerScrollRef.current.scrollTop = 0` fired at the real overflow container.
   - Console-gate.

Shared `beforeEach` sets up transparent-PNG image stubs and a route override that routes `f:commander` / `id<=` card-search queries to an inline `[solRing, forest]` fixture, while letting commander searches (`t:legendary`) and prints lookups (`unique=prints`) fall back to the standing `stubScryfall` auto-fixture.

All acceptance-criteria greps remained satisfied.

No Rule-1 / Rule-2 / Rule-4 deviations were required.

## Decisions Made

- **`.focus()` over Tab-count** for Tests 1 and 2: the commander panel re-renders on hydration and search results arrive async, so a deterministic `N-presses` approach is fragile. Direct focus exercises the same `:focus` styling paths the fixes target and is harness-stable.
- **Assert computed box-shadow (not just class presence)** for Tests 1 and 2: the per-component unit tests (`CommanderPanel.test.tsx`, `CardResultCell.test.tsx`) already pin class surface. The unique browser-level value is painted-pixel verification via `getComputedStyle(el).boxShadow !== 'none'` — this is what the class-level unit tests cannot give.
- **Synthetic overflow via 2000px spacer** for Test 3: the spec's deck has 0-2 cards; at default viewport the outer wrapper is never naturally scrollable. Injecting a tall spacer + seeding `scrollTop = 500` reproduces the exact UI-REVIEW Fix #2 scenario (user-scrolled, overflow-present) in a hermetic way.
- **No new fixtures, no helper changes** — reused `solRing` + `forest` fixtures already in the repo and the existing `createDeck` / `selectCommander` / `installConsoleGate` helpers.

## Functional Outcome

Running the suite now pins all three Phase 03.1 UI fixes against regression:

- Any future change that drops `focus:ring-2 focus:ring-accent` from CommanderPanel's action button → Test 1 fails on either class match OR computed box-shadow being `'none'`.
- Any future change that removes `tabIndex={0}` from CardResultCell OR drops the outer focus-ring classes → Test 2 fails on the `tabindex` attribute assertion OR the box-shadow assertion.
- Any regression that moves the view-toggle scroll-reset ref off the outer wrapper (or unwires `onViewToggle` from DeckWorkspace → DeckColumn) → Test 3's `expect.poll(...).toBe(0)` assertion fails.
