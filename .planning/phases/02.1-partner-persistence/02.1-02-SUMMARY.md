# Phase 02.1 Plan 02: Commander-Store Partner Persistence Summary

Wires the Zustand commander store's partner slot through Dexie v3 so the partner commander survives a hard reload: `setPartner`, `clearPartner`, `setCommander` auto-clear, and `clearCommander` all flush partner fields to the Deck row, and `loadForDeck` rehydrates both slots via Scryfall.

## What Shipped

- `setPartner(deckId, card)` and `clearPartner(deckId)` became async and persist to Dexie before updating Zustand state. Partner art is also `cacheCard`'d on set so a reload without network still paints the card.
- `setCommander` was restructured so the partner auto-clear branch merges `partnerCommanderId: null, partnerCommanderName: null` into the single `db.decks.update` call that already writes primary fields — one atomic write, no observer race.
- `clearCommander` now nulls both primary AND partner fields in the same Dexie update (matching the in-memory reset it already performed).
- `loadForDeck` reads `deck.partnerCommanderId`; if present it fetches the partner via `fetchCardById`, `cacheCard`s it, and sets `partnerCommander`. A Scryfall failure on the partner is caught locally so the primary still loads.
- `CommanderPanel.tsx` call sites updated to pass `deckId` and wrap the returned promises with `void`. This was mandatory: the method arity changed, so the type checker would have rejected the old call shape.
- Test suite: replaced the now-inverted "setPartner writes nothing" invariant, added Dexie assertions to `setCommander` and `clearCommander` partner paths, and added three brand-new tests including the flagship "partner survives reload" round-trip.

## Key Decisions

- **Arity change, not rename.** `setPartner` and `clearPartner` keep their method names (per ROADMAP direction "no store API renames") but expand arity to accept `deckId`. This is the minimum change required to persist, and the sole in-tree caller (CommanderPanel) was updated in the same commit.
- **Single-update compound writes.** The `setCommander` auto-clear path conditionally spreads partner-null fields into the existing `db.decks.update` call rather than issuing a second update. This avoids a transient window where an observer could see `commanderId` updated but stale `partnerCommanderId`.
- **Partner hydration is degradable.** `loadForDeck` wraps the partner `fetchCardById` in its own try/catch: a transient Scryfall 404 on a renamed partner card degrades gracefully to `partnerCommander: null` rather than aborting the primary load.
- **v2-shape rows read cleanly.** Legacy rows where `partnerCommanderId` is `undefined` short-circuit the partner fetch path — covered by a regression test so this stays a contract.
