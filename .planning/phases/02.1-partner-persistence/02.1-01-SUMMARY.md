# Phase 02.1 Plan 01: Partner Persistence Storage Seam Summary

Adds the typed storage seam (optional `partnerCommanderId` / `partnerCommanderName` on `Deck` plus Dexie version(3)) so plan 02 can persist the Zustand partner slot without breaking any v1 or v2 deck rows on disk.

## What Shipped

- `Deck` interface in `src/types/deck.ts` gained two optional nullable fields:
  `partnerCommanderId?: string | null` and `partnerCommanderName?: string | null`.
- `EDHBuilderDB` in `src/lib/db.ts` registers `this.version(3)` with the `decks` store schema string unchanged from v2 (`'++id, updatedAt'`) and no `.upgrade()` callback — Dexie handles the additive bump natively.
- `src/lib/db.test.ts` gained a new `describe('Dexie v3 migration')` block with three tests covering:
  1. v2-shaped rows read cleanly under v3 (partner fields come back `undefined`).
  2. Partner fields round-trip when populated.
  3. Explicit `null` updates persist as `null` (auto-clear path for plan 02).

## Key Decisions

- **Additive migration only.** No `.upgrade()` callback. Legacy rows are already valid v3 rows; missing keys read back as `undefined`. This mirrors the v1→v2 pattern recorded in STATE decisions.
- **Optional nullable typing.** `string | null | undefined` lets the store distinguish three states: never set (legacy), explicitly cleared (null, written by plan 02's auto-clear path), and populated (partner id string).

## Deviations from Plan

None — plan executed exactly as written. No auto-fixes (Rules 1–3) were triggered. No architectural decisions (Rule 4) surfaced.
