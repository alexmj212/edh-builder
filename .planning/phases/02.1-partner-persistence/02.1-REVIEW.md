## Summary

The partner-persistence implementation is architecturally sound: the Dexie v3 migration is purely additive, the store actions consistently flush to IndexedDB before updating in-memory state, and the component tests establish a genuine round-trip via fake-indexeddb. Three warning-level issues stand out: a primary-fetch failure in `loadForDeck` leaves a stale `partnerCommander` in state before the outer catch resets it â€” the reset path does not clear `partnerCommander`; the async `clearPartner` call in `CommanderPanel` is fire-and-forget with only two `Promise.resolve()` turns to confirm settlement in the test, making that assertion racey under slow fake-idb implementations; and `loadForDeck` silently swallows the partner hydration error without surfacing any diagnostic signal. Two info-level observations cover a type-width gap on `Deck` and a missing `afterEach` render-cleanup call in `CommanderPanel.test.tsx`.

## Findings

### WR-01: `loadForDeck` error path does not reset `partnerCommander` to null

**File:** `src/store/commander-store.ts:63`
**Severity:** warning
**Category:** bug

When `fetchCardById(deck.commanderId)` throws, the outer `catch` block runs:

```ts
set({ loading: false, error: (err as Error).message });
```

This does **not** set `partnerCommander: null`. Because `loadForDeck` opens by setting both to `null` on line 40, this is fine during a fresh load. However, if `loadForDeck` is called a second time on a deck that previously loaded successfully (e.g., deck switching), the store enters `loading: true` (which clears both fields per line 40), then the primary fetch throws, and the outer catch leaves `partnerCommander` in whatever state it had when the promise was created â€” which is `null` only because line 40 already reset it. This is currently safe but fragile: if the reset-on-entry line 40 is ever split or reordered, the error branch becomes a latent state inconsistency. Make the error path explicit:

```ts
set({ loading: false, error: (err as Error).message, primaryCommander: null, partnerCommander: null });
```

### WR-02: `clearPartner` async result in `CommanderPanel` is fire-and-forget; test timing is fragile

**File:** `src/components/CommanderPanel.test.tsx:122-130`
**Severity:** warning
**Category:** bug

The component calls `void clearPartner(deckId)` â€” intentional to avoid an unhandled promise in the JSX event handler. The test then asserts the Dexie row has been written after only two `await Promise.resolve()` microtask yields:

```ts
await Promise.resolve();
await Promise.resolve();
const deck = await db.decks.get(deckId);
```

`fake-indexeddb` can internally schedule IDB transactions across more than two microtask turns, especially when transaction queuing is involved. `waitForExpect` or `@testing-library/user-event` with `waitFor` would make this deterministic. As written the test passes under the current fake-idb implementation but could produce a false negative if the library's scheduling changes.

Suggested fix:

```ts
import { waitFor } from '@testing-library/react';
// ...
fireEvent.click(removeBtn);
await waitFor(async () => {
  const deck = await db.decks.get(deckId);
  expect(deck?.partnerCommanderId).toBeNull();
});
```

### WR-03: Partner hydration failure in `loadForDeck` is silently discarded with no diagnostic signal

**File:** `src/store/commander-store.ts:55-58`
**Severity:** warning
**Category:** maintainability

The inner `try/catch` around partner hydration correctly prevents a partner fetch failure from aborting the primary load. However it discards the error entirely:

```ts
} catch {
  // Partner hydration failure must not abort primary load.
  partner = null;
}
```

If the Scryfall ID stored in `partnerCommanderId` is stale or the network is degraded, the user sees the partner slot silently empty with no indication of the failure. The store has an `error` field that is already rendered (or will be). Preferably set a non-fatal warning, or at minimum log to the console so that during development the failure is visible:

```ts
} catch (err) {
  console.warn('[commander-store] partner hydration failed, clearing slot', err);
  partner = null;
}
```

### IN-01: `Deck.partnerCommanderId` / `partnerCommanderName` typed as `string | null | undefined` â€” optional fields widen unnecessarily

**File:** `src/types/deck.ts:7-8`
**Severity:** info
**Category:** maintainability

Declaring the fields as `?: string | null` (i.e., `string | null | undefined`) means callers can write three distinct states: absent key, explicit `undefined`, and `null`. The store only ever writes `null` (to clear) or a string (to set). The migration test at `db.test.ts:152` asserts `toBeUndefined()` for legacy v2 rows â€” which is correct for rows that predate the field. However, the three-value type forces every consumer to guard against both `undefined` and `null` separately, or use `??`. Consider narrowing to `string | null` (non-optional) with a default of `null` in new inserts, mirroring how `commanderId` and `commanderName` are typed. This would also make `db.decks.add` calls that omit the field a type error, which is actually the intent of the additive migration (existing rows are stored without the key, but new code should always supply it).

### IN-02: `CommanderPanel.test.tsx` does not call `cleanup()` between renders; relies on implicit RTL cleanup

**File:** `src/components/CommanderPanel.test.tsx:19-21`
**Severity:** info
**Category:** quality

The `afterEach` only calls `vi.restoreAllMocks()`. React Testing Library auto-registers a cleanup via `afterEach` when it detects the test framework (Vitest/Jest), so this is not a bug in current RTL versions. It becomes an issue if the project ever configures `@testing-library/react` with `configure({ asyncWrapper })` or moves to a manual setup that disables auto-cleanup. An explicit `import { cleanup } from '@testing-library/react'` and `afterEach(cleanup)` is cheap insurance and makes the intent clear.

## Notes

- The v3 schema block correctly omits an `upgrade` callback. Dexie's additive-only migration requires no callback when only non-indexed columns are added, and adding one that tries to backfill `partnerCommanderId: null` on every existing row would be incorrect (and unnecessary since Dexie reads missing fields as `undefined`). This is exactly right.
- `setCommander`'s conditional partner-field spread (`partnerFields`) is a clean pattern â€” it avoids writing `partnerCommanderId: undefined` into Dexie when the partner is preserved, which would accidentally overwrite a stored string with `undefined`.
- The `loadForDeck` test suite is well-structured: it explicitly covers the no-commanderId path, the no-partner path (v2-shape row), the partner-present path, and the reload-survives path. This is thorough coverage for the persistence round-trip.
- `db.test.ts` isolates v3 tests in their own `describe` block with its own `beforeEach`, keeping cross-version migration assertions independent. Good hygiene.
