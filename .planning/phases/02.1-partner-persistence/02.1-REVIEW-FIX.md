## Fixes Applied

### WR-01 — fixed

**Commit:** `efa1a21` — `fix(02.1): reset partner commander on loadForDeck error path`
**File:** `src/store/commander-store.ts`

Made the outer `catch` block in `loadForDeck` explicitly reset both `primaryCommander` and `partnerCommander` to `null` alongside the existing `loading` / `error` fields. Previously the reset relied on the entry-time `set({ ..., primaryCommander: null, partnerCommander: null })` call, which is a latent inconsistency if that entry reset is ever reordered or split.

```diff
     } catch (err) {
-      set({ loading: false, error: (err as Error).message });
+      set({
+        loading: false,
+        error: (err as Error).message,
+        primaryCommander: null,
+        partnerCommander: null,
+      });
     }
```

### WR-03 — fixed

**Commit:** `d5dbb38` — `fix(02.1): surface partner hydration failures via console.warn`
**File:** `src/store/commander-store.ts`

Replaced the bare `catch {}` on partner hydration with `catch (err)` plus a `console.warn` that includes the underlying error. The primary-load behavior is unchanged (partner still clears to `null`), but a stale `partnerCommanderId` or network blip is now visible during development instead of silently rendering an empty partner slot.

```diff
-        } catch {
+        } catch (err) {

+          console.warn('[commander-store] partner hydration failed, clearing slot', err);

```
